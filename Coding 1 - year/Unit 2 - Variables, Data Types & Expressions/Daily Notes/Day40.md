# Day 40: Mini-Project Presentations and Peer Review

**Learning Target:** I can present my program clearly, explain my design choices, and provide constructive feedback to peers.

### Purpose

Today is about celebrating the work you've done and learning from your classmates. You'll present your calculator program and give/receive feedback on code quality and functionality.

### Presentation Guidelines (2-3 minutes per project)

**What to include:**
1. **Introduction:** Name your project and explain what it does in one sentence
2. **Demo:** Run your program with sample inputs and show the output
3. **Code Walkthrough:** Show 2-3 interesting parts of your code and explain them
4. **Design Choices:** Explain one choice you made (Why that calculation? Why that format?)
5. **Challenges:** What was hardest about building this?

**Example Introduction:**
"My project is a Pizza Party Cost Calculator. It helps split the bill when you order pizza with friends. You tell it how many pizzas, how many people, and the tax rate, and it shows the cost per person."

### Peer Review Feedback Form

For each project you watch, complete a feedback form:

**Strengths:** (What did they do well?)
- [ ] Program runs without errors
- [ ] Clear variable names
- [ ] Good use of constants
- [ ] Professional output formatting
- [ ] Interesting or useful program

**Suggestions for Improvement:**
- Could variable names be more descriptive?
- Could the output be formatted better?
- Are there calculations that could be simplified?
- Would input validation help?
- Could the program do more?

**One Compliment:**
"I really liked how you [specific thing]."

**One Question:**
"How would you [extend the program / handle a different case]?"

### Self-Reflection

After your presentation, reflect on your own project:

1. **What went well?**
   - What part of the program are you most proud of?
   - What was easiest to implement?

2. **What was challenging?**
   - What gave you trouble?
   - How did you solve it?

3. **If you had more time, what would you add?**
   - More calculations?
   - Better validation?
   - More features?
   - A menu system?

4. **What did you learn from classmates' projects?**
   - Did anyone use a technique you hadn't thought of?
   - Did any project inspire ideas for your own code?

### Audience Expectations

As an audience member:
- [ ] Listen actively and take notes
- [ ] Ask at least one question during Q&A
- [ ] Provide one piece of constructive feedback
- [ ] Congratulate the presenter

### Common Questions You Might Get Asked

**Q: How would you modify this for [different scenario]?**
Think about how your program could adapt. Be ready to discuss.

**Q: Why did you choose to [design choice]?**
Be ready to explain your reasoning. There's usually not a "wrong" answer, just tradeoffs.

**Q: What was the hardest part?**
Be honest. Learning happens through challenges!

**Q: Can you show me the code for [feature]?**
You should be able to point to your code and explain key sections.

**Q: What would you do differently next time?**
Reflection shows maturity. It's okay to identify ways to improve.

### Fun Facts to Share

As you present, consider sharing:
- How long it took to write
- A bug you found and fixed
- An interesting calculation in your program
- Why you chose this particular calculator

### What Happens After Presentations

1. Save your code (you'll turn it in with Unit 2 Assessment)
2. Note feedback from classmates
3. Optional: Improve your project based on feedback
4. Reflect on what you learned
