# Day 116: Mini-Project Presentations and Peer Code Review

**Learning Target:** I can present my code, explain design decisions, review others' code, and provide constructive feedback.

### Overview

Today is celebration day and learning day! You'll **share your mini-projects** with the class, **receive feedback**, and **review others' code**. This teaches essential professional skills: communication, code reading, and collaborative improvement.

### Presentation Guidelines

**When presenting (3–5 minutes):**

1. **The Problem** — What does your program do? What problem does it solve?
2. **The Design** — How did you break it into functions? Why that design?
3. **Key Functions** — Show 1–2 interesting functions. Explain what they do.
4. **A Demo** — Run your program and show it working.
5. **Lessons Learned** — What was challenging? What would you do differently?

**Tips:**
- Don't read your code verbatim—explain the ideas
- Focus on design, not syntax details
- Highlight your best work
- Be ready to answer questions

### Code Review Guidelines

**When reviewing someone else's code:**

**Read through the code and look for:**

1. **Clarity** — Can you understand what the functions do? Are names clear?
2. **Organization** — Is the code well-structured? Does main() make sense?
3. **Documentation** — Are there docstrings and comments where needed?
4. **Scope** — Does the code use local variables appropriately? Avoid globals?
5. **Bugs** — Do you see any potential issues?
6. **Good practices** — What did they do well?

**Provide feedback using this format:**

```
Program: [Name]
Presenter: [Name]

STRENGTHS (What they did well):
- [Thing 1]
- [Thing 2]

SUGGESTIONS (What could improve):
- [Suggestion 1]
- [Suggestion 2]

QUESTIONS (What you're curious about):
- [Question 1]
```

**Keep feedback constructive:**
- Start with something positive
- Be specific, not vague
- Offer improvement ideas, not just criticism
- Ask questions to understand their choices

### Code Review Checklist

For each program, ask:

- [ ] Can I understand what each function does without reading the implementation?
- [ ] Does the program have a clear main() function?
- [ ] Are variable names clear and descriptive?
- [ ] Does the code avoid unnecessary global variables?
- [ ] Can I find at least one example of function composition (functions calling functions)?
- [ ] Does the program handle edge cases reasonably?
- [ ] Is the code well-commented?
- [ ] Would I be able to modify or extend this program?

### Presentation Order

Your teacher will assign presentation order. Typically:
1. Presenter introduces their program
2. Presenter runs a demo
3. Audience asks questions (2–3 min)
4. Code review partners provide written feedback
5. Move to next presenter

### What You'll Learn

Watching others present:
- **Different design approaches** — See how different people solve the same problem
- **Code reading skills** — Learn to understand code written by others
- **Best practices** — Notice what makes code clear and maintainable
- **Feedback skills** — Learn to give and receive constructive criticism

### After Presentations

1. **Take notes** on feedback you receive
2. **Thank reviewers** for their input
3. **Reflect** on what you'd improve
4. **Save feedback** for future reference

### Self-Review Questions

Before presenting, ask yourself:

1. Can I explain the purpose of each function in one sentence?
2. Can I explain my design choices to the class?
3. Are there any bugs or edge cases I didn't handle?
4. If someone else read my code, would they understand it?
5. What's one part I'm proud of? Why?

### Evaluation Rubric (for reference)

Your presentation and code will likely be evaluated on:

| Criterion | Excellent | Good | Fair |
|-----------|-----------|------|------|
| **Design** | Clear functions, good composition | Logical structure, most choices make sense | Some organization, unclear choices |
| **Code Quality** | Well-documented, clean, clear | Mostly clear with some rough spots | Hard to read, minimal comments |
| **Functionality** | Works well with multiple test cases | Works for main cases, minor bugs | Limited functionality, significant bugs |
| **Presentation** | Clear, confident, good demo | Explains code, shows it works | Unclear explanation or demo issues |
| **Documentation** | Excellent docstrings and comments | Good docstrings, adequate comments | Minimal documentation |
