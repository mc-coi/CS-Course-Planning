# Unit 1 - Introduction to Programming: Pacing & Differentiation Guide

## Unit Overview
- **Total days allocated:** 20 days / 4 weeks (55-minute periods, traditional schedule)
- **Core topics covered:** Computer science fundamentals, algorithms, Python syntax (print, variables, data types, input, arithmetic, formatting, type conversion), software development process
- **Prerequisites students need:** Basic computer skills; familiarity with a text editor or IDE; understanding of place value and basic arithmetic

## Week-by-Week Pacing

### Week 1 (Days 1–5): What is CS & Algorithms
**Topics:** Computer science as problem-solving, algorithms (definition, properties), pseudocode, flowcharts, intro to Python and print()

#### Essential days (cannot skip):
- **Day 1:** Computer Science definition and algorithms concept (foundational—all later units depend on this understanding)
- **Day 3:** Python setup and first print() program (must get students coding)
- **Day 5:** Practice lab (allows practice with print, comments, escape characters before moving forward)

#### Flexible days (can compress or cut if pressed for time):
- **Day 2 (Pseudocode & Flowcharts):** If time is tight, this can be abbreviated to 15 minutes of explanation with one quick example, rather than a full lesson. Students can still learn the concept of planning without extensive practice.
- **Day 4 (Comments & Print Formatting):** Can combine with Day 3 or push some content to Day 5's practice lab

#### If ahead of pace:
- Have students write pseudocode for more complex algorithms (sorting, searching)
- Introduce drawing flowcharts for algorithms
- Challenge them to modify print() examples with complex escape character sequences
- Have them teach pseudocode/flowchart concepts to a partner

#### If behind pace:
- Spend extra time on Day 3 (Python setup) if students struggle with IDE or syntax
- Shorten Day 2 pseudocode lesson—focus on algorithm definition only, defer flowcharts
- Combine Days 4 and 5 into one extended lab day
- **Minimum viable coverage:** Days 1, 3 (Python basics), and 5 (practice)

---

### Week 2 (Days 6–10): Variables & Data Types
**Topics:** Variable naming rules, assignment, reassignment, four data types (str, int, float, bool), input(), string concatenation

#### Essential days (cannot skip):
- **Day 6:** Variables (naming rules, assignment, reassignment) — core concept all future code depends on
- **Day 7:** Data types and type() function — students must understand the four types
- **Day 8:** input() and string concatenation — essential for interactive programs
- **Day 10:** Practice lab (solidifies all three days' concepts)

#### Flexible days (can compress or cut if pressed for time):
- **Day 9 (Type Conversion):** Can be introduced via quick demo (5 min) with practice problems, rather than a full lesson. Students often learn this better through problem-solving anyway.

#### If ahead of pace:
- Introduce the type() function more deeply
- Have students predict data types before running code
- Introduce variable naming conventions (snake_case, meaningful names)
- Challenge: ask them to write a program that collects three pieces of different-type data and displays them

#### If behind pace:
- Spend extra time on Day 6 and 7 (variables and types are foundational)
- Shorten Day 9 type conversion—practice only the most common conversions (int(), str())
- Combine Days 8 and 9 into one day focused on practical input/concatenation problems
- **Minimum viable coverage:** Days 6–8 (variables, types, input)

---

### Week 3 (Days 11–15): Basic Calculations & Expressions
**Topics:** Arithmetic operators (+, -, *, /, //, %, **), order of operations (PEMDAS), arithmetic with variables, f-string formatting

#### Essential days (cannot skip):
- **Day 11:** Arithmetic operators — students need to know all six operators
- **Day 12:** Order of operations (PEMDAS) — a frequent source of bugs; must be explicit
- **Day 13:** Math with variables — combining operators and variables in real programs
- **Day 14:** F-strings — modern, professional output formatting (students will use this constantly)
- **Day 15:** Practice lab (calculator programs, unit converters)

#### Flexible days (can compress or cut if pressed for time):
- None. All five days are important for building calculator-style programs, which are the capstone for this unit.

#### If ahead of pace:
- Introduce exponentiation and modulo with deeper examples (prime checking, digit extraction)
- Have students calculate more complex formulas (compound interest, BMI, distance)
- Introduce comments for explaining complex expressions
- Challenge: write a program with mixed operations and multiple f-string outputs

#### If behind pace:
- Spend extra time on Day 12 (PEMDAS)—many students struggle with operator precedence
- Use simpler formulas on Days 13–15 (avoid compound interest; stick to area, distance, etc.)
- Reduce the number of practice problems assigned; focus on depth over breadth
- **Minimum viable coverage:** Days 11–14 (operators, order of operations, f-strings)

---

### Week 4 (Days 16–20): Software Development & Review
**Topics:** 5-step development process, debugging, syntax and type errors, mini-project, unit review, assessment

#### Essential days (cannot skip):
- **Day 16:** Software Development Process (planning, coding, testing, refining) — essential mindset for all future projects
- **Day 18:** Mini-project — students must create one complete program from scratch
- **Day 20:** Unit Assessment — formal evaluation

#### Flexible days (can compress or cut if pressed for time):
- **Day 17 (Debugging Intro):** If time is short, cover syntax and type errors only (runtime and logic errors can be deferred to Unit 6)
- **Day 19 (Unit Review):** Can be shortened to 15 minutes of Q&A rather than a full review lesson

#### If ahead of pace:
- Have students plan and code more complex mini-projects (multiple interacting features)
- Introduce more types of errors and debugging strategies
- Have them debug code written by peers
- Challenge: write a program with intentional bugs for others to find

#### If behind pace:
- Shorten Days 17 and 19 to quick reviews only
- Allow Day 18 mini-project to be extended into a take-home assignment (rather than in-class only)
- Reduce complexity of mini-project requirements
- **Minimum viable coverage:** Days 16, 18 (process, mini-project) and a quick assessment

## Differentiation Strategies

### For Struggling Students
**Specific scaffolding tips:**
- **Variables & Types:** Use real-world analogies (a variable is a labeled box; data types are different kinds of things you put in the box). Have them sort objects by type before coding.
- **Order of Operations:** Provide a PEMDAS anchor chart. Have them solve expressions by hand first, then verify with code.
- **Type Conversion:** Focus on practical examples (convert score to string to concatenate with text). Avoid abstract explanations.
- **Print & Input:** Start with simple programs (ask name, print greeting) before moving to multi-step programs.

**Which practice problems to assign first (easiest wins):**
- From UnitPractice.md, start with Beginner problems 1-6 (print, variables, simple input)
- Then Intermediate 11-12 (simple conversions, one operation)
- Build confidence before attempting Intermediate 13+ (multi-step calculations)

**Suggested pacing adjustments:**
- Spend 2 days on variables instead of 1 (Day 6 split into two half-days)
- Spend 2 days on order of operations (extend Day 12)
- Provide worked examples for every problem type before asking students to solve independently
- Allow use of calculator for checking arithmetic during lessons

**What prerequisite gaps to look for:**
- **Place value & number concepts:** Ask "What does the 5 represent in 25?" If confused, they'll struggle with operators
- **Basic algebra:** Can they solve "3x + 2 = 11"? If not, expressions with variables will confuse them
- **Reading comprehension:** Do they understand what a problem is asking? Read problems aloud; have them rephrase

### For On-Track Students
**Standard path through the unit:**
- Follow the week-by-week breakdown as written
- Complete the Beginner and Intermediate practice problems
- Focus on building real programs (calculator, unit converter) rather than isolated exercises

**Which practice problems to assign:**
- Beginner (1-10) for foundational skill-building
- Intermediate (11-20) for synthesis and application
- Select 2-3 Challenge problems (21-30) for stretching understanding
- Daily practice labs (Days 5, 10, 15, 18) are essential

**Expectations:**
- Code should have comments explaining key steps
- Variables should be meaningfully named (not x, y, z)
- All programs should be tested with multiple inputs

### For Advanced Students
**Extension activities specific to this unit's topics:**
- **Algorithms:** Have them write pseudocode and flowcharts for real-world processes (making breakfast, playing a game, sorting a list)
- **Type system:** Explore edge cases (what happens with very large numbers? very small decimals?) and type coercion
- **String operations:** Teach string indexing and slicing (from Unit 2 preview) so they can manipulate strings creatively
- **Math module:** Introduce `import math` and functions like `math.sqrt()`, `math.pi`, `math.floor()`

**Challenge problems to assign:**
- From UnitPractice.md, assign Challenge problems 21-30 (compound interest, student profile, BMI, time calculations)
- Have them create a "program from scratch" that combines 3+ operators and user input
- Ask them to debug intentionally broken code and explain all three types of errors found

**Ways to deepen understanding beyond the curriculum:**
- Have them explain why Python's `/` always returns a float (deep understanding of division)
- Explore truthy/falsy values (`bool("")`, `bool(0)`) — preview of Unit 3
- Write a program that validates user input and re-asks if invalid (preview of Unit 3 conditionals and Unit 4 loops)
- Create a "Tips & Tricks" document for their future selves about common mistakes in this unit

## Flexibility Notes

### Natural pause points for re-teaching:
- **End of Week 1:** Before introducing variables, make sure students have the algorithm mindset
- **End of Week 2:** Before moving to arithmetic, ensure students are comfortable with variables and basic input/output
- **Day 13/14:** After students complete calculator programs, this is a good checkpoint to assess before moving to Week 4

### Topics students most commonly struggle with (and expected time):
1. **Order of Operations (PEMDAS):** 1-2 extra days may be needed. Many students compute left-to-right instinctively.
2. **Type Conversion:** 1 extra day. Students often forget int() or str() is needed. Practice with "fixing broken code" helps.
3. **String Concatenation vs. Arithmetic:** 1 extra day. The error `"Score: " + 42` confuses many; use this as a teachable moment.
4. **F-strings:** 1-2 days. Modern syntax; worth mastering now (used in all future units).
5. **Variable Naming:** 30 minutes. Explain why x, y, z are bad; meaningfulness matters.

### Topics students typically move through quickly:
- **Comparison operators:** Once they see `<` and `>`, they grasp it in 5 minutes
- **Basic print() statements:** Most students pick this up in the first day
- **Simple arithmetic:** If they know basic math, operators feel intuitive
- **Input and concatenation:** Usually clicks once they see one full example

## Suggested Checkpoints

Use these quick formative checks mid-unit (not full assessments, just quick pulse checks):

### After Day 5 (end of Week 1):
- **Quick Check:** "Write pseudocode for 'make a sandwich.'" Are steps precise and ordered?
- **Code Check:** Can they write `print("Hello")` and add a comment?

### After Day 10 (end of Week 2):
- **Quick Check:** "Declare a variable for your age. What data type is it?" (int)
- **Code Check:** Write a program that asks the user's name and prints a greeting using concatenation

### After Day 15 (end of Week 3):
- **Quick Check:** "What is 3 + 2 * 4?" (Answer: 11, not 20) — validates PEMDAS understanding
- **Code Check:** Write a program that calculates the area of a rectangle (length × width) using f-strings to format output

### Before Day 16 (mid-Week 4):
- **Quick Check:** "What are the 5 steps of the development process?" (Define, Plan, Code, Test, Refine)
- **Code Check:** Students hand in their mini-project. Quick scan: Does it run? Does it solve the stated problem?

## Common Misconceptions to Watch For

1. **"Variables store the code that made them"** — Students think if they write `x = 5 + 3`, x contains "5 + 3". Correct: x contains only 8.
2. **"= is the same as =="** — They'll use `if x = 5:` in Unit 3. Emphasize: = assigns, == compares. Use this language consistently now.
3. **"Division always gives a whole number"** — In Python 3, `/` returns float. Show 10/2 = 5.0 on day 1 of arithmetic.
4. **"You can change a string by modifying an index"** — "Hello"[0] = "J" will error. Preview that strings are immutable (Unit 2 topic).
5. **"Type conversion is optional"** — `"Score: " + str(42)` is necessary. Show the error when you forget str().

## Notes on Assessment

- **Unit 1 Assessment (Day 20):** Should include short-answer (What is an algorithm?), code tracing (What does this print?), and a mini-coding task (write a program that...). This mirrors the mini-project work.
- **Retakes:** If a student doesn't pass, offer a retake focused on areas of weakness (e.g., if they struggle with type conversion, focus the retake there).
