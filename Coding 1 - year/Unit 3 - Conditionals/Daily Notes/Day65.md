## Presentation Guidelines

### What to Show (5-7 minutes per project)

**1. Program Overview (1 minute)**
- What does your program do?
- Who is it for?
- What problem does it solve?

**2. User Experience Walkthrough (2-3 minutes)**
- Run the program
- Show normal usage (happy path)
- Show what output looks like
- Demonstrate at least 2 different scenarios

**3. Code Highlights (2 minutes)**
- Show most interesting conditional logic
- Explain a complex decision
- Point out input validation
- Mention challenges overcome

**4. Q&A (1 minute)**
- Answer questions from audience

---

## Presentation Tips

**Good Presentations:**
- Speak clearly; face the audience
- Run program successfully
- Explain what's happening (not just running code)
- Use examples that show different paths
- Answer questions thoughtfully

**Avoid:**
- Reading code line-by-line
- Assuming audience knows the code
- Rushing through explanations
- Not testing before presenting
- Leaving output unclear

## Peer Review Template

### Functionality
- [ ] Program runs without errors
- [ ] All features work as described
- [ ] Handles different inputs correctly
- [ ] Provides clear output

### Code Quality
- [ ] Logic is easy to follow
- [ ] Variable names are clear
- [ ] Comments explain complex parts
- [ ] Indentation is consistent

### Completeness
- [ ] Meets all requirements
- [ ] Has at least 5 decision branches
- [ ] Uses conditionals effectively
- [ ] Shows good problem-solving

### User Experience
- [ ] Program is easy to use
- [ ] Prompts are clear
- [ ] Feedback is helpful
- [ ] Edge cases handled

### Overall Assessment
- **Strengths:** (What did the student do well?)
- **Growth Areas:** (What could be improved?)
- **Questions:** (Anything you'd like to know more about?)

---

## Constructive Feedback Guidelines

**Good Feedback:**
- "Your menu system was very clear. The options were easy to understand."
- "The error handling was strong. It caught invalid inputs well."
- "I liked how you explained your decision logic. It made it easy to follow."
- "You could enhance this by adding X feature..."
- "What made you choose that approach for Y?"

**Avoid:**
- "Your code is bad"
- Negative comments without suggestions
- Comparing to other students
- Personal criticism
- Comments that aren't constructive
