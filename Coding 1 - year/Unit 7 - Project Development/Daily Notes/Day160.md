## Key Concepts

### Sprint Retrospective

A **retrospective** is a meeting where a team reflects on what went well and what could improve. You'll do this solo (or with your peer).

**Questions to ask:**
1. What worked well?
2. What was harder than expected?
3. What will we change next sprint?
4. What should we keep doing?

Day 160 is also about celebrating progress! You've built:
- ✓ Project structure
- ✓ Core functions

- ✓ Code review and improvement

That's significant!

## Guided Practice

### Your Retrospective (Full Period)

**Step 1: Assess What You've Built (10 min)**
- Count your functions
- List what works
- Identify any remaining bugs
- Celebrate what you built!

**Step 2: Reflect on Process (10 min)**
- What made coding easy?
- What made coding hard?
- When did you feel stuck?
- How did you get unstuck?

**Step 3: Evaluate Code Quality (10 min)**
- Check docstrings — are all functions documented?
- Check readability — are variable names clear?
- Check organization — is code well-structured?
- Check testing — did you test edge cases?

**Step 4: Identify Improvements (10 min)**
- Based on Day 159 code review, what will you fix?
- Any refactoring needed?
- What testing will you add?

**Step 5: Plan Sprint 2 (10 min)**
- What features are left?
- In what order should you build them?
- What's your priority?
- How many days do you have?

**Step 6: Commit & Document (5 min)**
- Write your retrospective
- Commit with message: "Day 160: Sprint 1 complete, retrospective done"
- Save retrospective in a document
