# Day 147: Writing a Project Proposal

**Learning Target:** I can write a clear project proposal including problem statement, target audience, core features, and success criteria.

---

## Key Concepts

### The Project Proposal

Your **proposal** is your contract with yourself. It answers:
- What are you building?
- Why does it matter?
- Who will use it?
- What will it do?
- How will you know when it's done?

A good proposal is **1-2 pages** and takes **30-45 minutes** to write. It saves hours of wasted coding.

### Elements of a Strong Proposal

**1. Project Title**
- Clear and specific
- Example: "Grade Tracker for High School Students" (not just "Grade App")

**2. Problem Statement**
- What problem does it solve? (1-2 sentences)
- Who experiences this problem? (1-2 sentences)
- Why is it important? (1 sentence)
- Example: "High school students struggle to track grades across multiple classes. Calculating averages manually is tedious and error-prone. A tool that automatically tracks and calculates would save time and reduce stress."

**3. Target Audience**
- Who are your users?
- What do they need?
- Example: "High school students ages 14-18 who want to monitor their academic progress throughout the year."

**4. Core Features (Must-Have)**
- The MVP (Minimum Viable Product)
- Usually 3-5 features
- Examples:
  - Add and store student scores
  - Calculate grade averages
  - Assign letter grades (A-F)
  - Save data to a file

**5. Nice-to-Have Features (Optional)**
- Things you'd add if you have time
- Examples:
  - Track grades by subject separately
  - Predict final grade based on current average
  - Compare performance over time

**6. Technology & Data Structures**
- What Python features will you use?
- Example: "Lists to store scores, dictionaries to organize by subject, file I/O to save data, loops and conditionals for logic"

**7. Success Criteria**
- How will you know it's done?
- Examples:
  - User can add scores without the program crashing
  - Program handles invalid input gracefully
  - Grades are calculated correctly
  - Data persists to file and loads on restart

## Code Examples

### Example Proposal: Grade Tracker

```
PROJECT TITLE: Grade Tracker

PROBLEM STATEMENT:
High school students struggle to keep track of their grades across multiple
classes throughout the year. Calculating grade averages by hand is tedious and
error-prone, especially when tracking multiple subjects. An automated grade
tracker would save time, reduce calculation errors, and help students monitor
their academic progress.

TARGET AUDIENCE:
High school students (grades 9-12) who want to track grades in multiple classes
and understand their academic standing without manual calculation.

CORE FEATURES (MVP):
1. Add student names and their scores for a class
2. Automatically calculate grade averages
3. Assign letter grades (A-F) based on numeric averages
4. Display a formatted report showing all scores, average, and grade
5. Save reports to a file for later reference

NICE-TO-HAVE FEATURES:
1. Track grades separately by subject
2. Predict what final grade is needed to reach a target
3. Compare current average to previous marking period
4. Generate a class-wide statistics report (not just individual)

TECHNOLOGY & DATA STRUCTURES:
- Lists: Store scores for a student
- Dictionaries: Store student names → scores mapping (optional)
- Functions: Separate functions for getting input, calculating, displaying
- File I/O: Read/write reports to text files
- Error handling: try/except for invalid input and file operations
- Loops: For entering scores, calculating statistics

INPUTS & OUTPUTS:
Inputs: Student name, scores (via user input and/or file)
Outputs: Printed reports, saved text files

SUCCESS CRITERIA:
1. User can enter multiple scores without the program crashing
2. Calculations are accurate (tested with known values)
3. Program handles invalid input (letters instead of numbers) gracefully
4. Reports are formatted clearly and easy to read
5. Data can be saved to a file and retrieved later
6. All functions have docstrings and code is well-commented

TEST CASES:
- Normal: Enter 3 scores (85, 92, 78) → average = 85.0, grade = B
- Edge case: Enter 0 scores → appropriate error message
- Edge case: Enter 1 score → average = that score
- Error case: User enters "abc" instead of number → program asks again
- Error case: File doesn't exist when loading → appropriate error message
```

## Guided Practice

### Write Your Proposal (30-45 minutes)

Using the template above, write a proposal for your capstone project. Follow this process:

**Step 1: Choose Your Project (5 min)**
- Pick from the suggestions in Unit 7, or create your own
- It should use most of what you learned (lists, functions, loops, file I/O, etc.)
- It should interest you (you'll spend 5 weeks on this!)

**Step 2: Problem Statement (10 min)**
- Write 1-2 sentences: What problem does it solve?
- Write 1-2 sentences: Who experiences this problem?
- Write 1 sentence: Why is it important?

**Step 3: Core Features (5 min)**
- List 3-5 must-have features (the MVP)
- Skip the nice-to-haves for now — focus on what's essential

**Step 4: Success Criteria (10 min)**
- How will you know this project is "done"?
- Include: functionality (what it does), code quality (clean and documented), testing (works on edge cases)

**Step 5: Get Feedback (5 min)**
- Share with a peer or your teacher
- Ask: "Is this clear? Could I build this in 4 weeks?"

## Practice Problems

**Problem 1 — Proposal Review:** Read the Grade Tracker proposal above. For each core feature, explain what function(s) you'd need to implement it.

**Problem 2 — Define Your Project:** Write a 1-paragraph problem statement for a project *you* want to build. Include: what problem it solves, who uses it, why it matters.

**Problem 3 — Challenge:** Your proposal includes 5 core features. You only have time to implement 3 before the first code review. Which 3 would you do first, and why?
