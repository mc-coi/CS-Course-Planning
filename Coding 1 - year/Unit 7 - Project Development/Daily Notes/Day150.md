# Day 150: Peer Review of Project Proposals

**Learning Target:** I can give and receive constructive feedback on project proposals, and refine my own proposal based on feedback.

---

## Key Concepts

### What is Peer Review?

**Peer review** means having a classmate or colleague read your work and give honest, constructive feedback. It's a professional practice used in software development, academia, and publishing.

**Benefits:**
- Fresh eyes catch problems you missed
- You get ideas for improvements
- It builds communication skills
- It's faster than discovering problems in code later

### Good Feedback vs. Bad Feedback

**Bad feedback:** "Your project is cool!" or "This sucks."
(Vague, not helpful)

**Good feedback:** "I'm not sure what 'analytics' means — can you be more specific? Does that mean showing trends over time? If so, how will you calculate trends?"
(Specific, actionable, constructive)

### Peer Review Checklist

When reviewing someone else's proposal, ask:

**Clarity:**
- [ ] Is the problem statement clear? Can I explain it back?
- [ ] Are the core features understandable?
- [ ] Is there anything vague or confusing?

**Feasibility:**
- [ ] Can this be built in 4 weeks?
- [ ] Do they have the skills needed?
- [ ] Are the core features reasonable (not too simple, not too complex)?

**Completeness:**
- [ ] Does the proposal answer: what? why? who? how?
- [ ] Are success criteria clear?
- [ ] Is the technology appropriate?

**Scope:**
- [ ] Is the MVP appropriate (not too big, not too small)?
- [ ] Are the nice-to-haves truly optional?

## Guided Practice

### Peer Review Session (Full Period)

**Phase 1: Exchange Proposals (5 min)**
- Partner up with a classmate
- Exchange proposals (physical or digital)
- Everyone reads their partner's proposal carefully

**Phase 2: Ask Clarifying Questions (15 min)**
- For each major section, ask: "What do you mean by...?"
- Don't assume — ask for specifics
- Examples:
  - "You mention 'user authentication.' Does that mean username and password, or just a username?"
  - "What data structure will you use to store the todo items?"
  - "What happens if the user types invalid input?"

**Phase 3: Give Constructive Feedback (10 min)**
- Use this format:
  1. "What I like about your project: [positive thing]"
  2. "One question I have: [clarifying question]"
  3. "One suggestion: [constructive improvement]"

**Phase 4: Listen and Note Feedback (10 min)**
- Your partner gives you feedback
- Don't defend or explain — just listen
- Write down their feedback (even if you disagree)
- Thank them

**Phase 5: Plan Revisions (5 min)**
- What feedback will you incorporate?
- What will you leave as-is?
- Note: You don't have to implement every suggestion, but think about why

## Code Examples

### Example: Reviewing a Proposal

**Original Proposal (Vague):**
```
PROJECT TITLE: Music App

PROBLEM STATEMENT:
Users want to listen to music. My app helps them do that.

CORE FEATURES:
1. Play music
2. Organize music
3. Make playlists
4. Search songs

TEST CASES:
None listed
```

**Feedback:**
- **Question 1:** "When you say 'play music,' do you mean the user provides audio files? Or do you fetch them from the internet?"
- **Question 2:** "How will users 'organize' music? By artist? Genre? Custom folders?"
- **Question 3:** "Your test cases are missing. What will you test?"
- **Suggestion:** "Consider starting with just playing local music files and creating playlists. That's a full MVP."

**Revised Proposal:**
```
PROJECT TITLE: Local Music Player & Playlist Manager

PROBLEM STATEMENT:
Users have music files on their computer but lack a simple tool to organize and
play them. They want to create playlists and quickly find songs. My app provides
an easy way to manage local audio files and create custom playlists.

CORE FEATURES:
1. Load and list all .mp3 files from a folder
2. Display song metadata (filename, duration, artist if available)
3. Play a selected song (using Python's built-in audio or a library)
4. Create and save playlists (list of song filenames to a text file)
5. Load saved playlists
6. Search for songs by name

NICE-TO-HAVE:
1. Sort songs by artist or duration
2. Play entire playlist in order
3. Remove songs from playlist

TEST CASES:
- Folder with 0 songs → show "No songs found"
- Folder with 5 .mp3 files → display all 5 correctly
- Create playlist with 3 songs → save and load playlist, verify songs appear
- Search for song → correctly find matching songs
- Invalid folder path → show error message
```

## Practice Problems

**Problem 1 — Review Practice:** Read this proposal snippet and identify three questions you would ask:

```
PROJECT: Todo List App
PROBLEM: People need to track tasks
CORE FEATURES: Add tasks, delete tasks, see tasks
```

**Problem 2 — Feedback Evaluation:** Your partner says, "Your project is too big. You should make it simpler." Is this helpful feedback? How would you respond?

**Problem 3 — Self-Review:** Review your own proposal as if you were a peer. What's unclear? What could be better? Write 2-3 improvements.

<details>
<summary>💡 Hints</summary>

- Problem 1: "Track tasks" is vague. "Too big" is vague. Find the unclear parts and ask specific questions.
- Problem 2: "Too big" is not helpful. You could ask: "Which features feel unnecessary? Which ones depend on others?" Then decide.
- Problem 3: Read your proposal as if you've never heard of the project. What would confuse you?

</details>

<details>
<summary>✅ Sample Feedback Questions</summary>

For the Todo List App above, good questions would be:
1. "What happens if the user closes the app — do tasks get saved? How?"
2. "What does 'see tasks' mean — do you display them in a list? Do you show status (done/not done)?"
3. "What counts as a 'task'? Just a name, or does it include due date, priority, etc.?"

</details>
