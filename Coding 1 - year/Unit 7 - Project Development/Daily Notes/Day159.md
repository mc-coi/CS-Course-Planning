# Day 159: Code Review and Collaborative Debugging

**Learning Target:** I can participate in code review to improve code quality and solve difficult bugs collaboratively.

---

## Key Concepts

### Code Review Process

**Code Review** means: Another programmer reads your code and provides feedback.

**It's not:**
- Criticism of you personally
- Looking for perfect code
- Blaming bugs on the programmer

**It is:**
- Improving code quality together
- Finding bugs before they cause problems
- Learning from each other
- Professional best practice

### What to Look For in Code Review

**Functionality:** Does the code do what it's supposed to?
- [ ] Function produces correct output
- [ ] Edge cases handled
- [ ] No crashes on bad input

**Readability:** Can someone else understand the code?
- [ ] Variable names are clear
- [ ] Functions are short and focused
- [ ] Logic flow is obvious
- [ ] Docstrings explain purpose

**Structure:** Is the code organized well?
- [ ] Related functions grouped
- [ ] No repeated code (DRY principle)
- [ ] Good use of functions (not monolithic)

**Best Practices:**
- [ ] Error handling with try/except
- [ ] Meaningful comments (why, not what)
- [ ] Consistent style

## Collaborative Debugging

When you're stuck on a bug together:

**1. Describe the problem (2 min)**
```
"When I enter spaces for a name, the program crashes."
```

**2. Identify the error (3 min)**
```python
# Peer tries it
name = input("Name: ").strip()
if not name:  # This only checks if empty after stripping
    # But what if input is "   " (just spaces)?
```

**3. Brainstorm solutions (3 min)**
```
Option A: name = input("Name: ").strip(); if not name: ...
Option B: name = input("Name: ").strip(); if len(name) == 0: ...
Option C: name = input("Name: ").strip(); if not name.isalpha(): ...
```

**4. Test the fix (2 min)**
```python
test_input = "   "
stripped = test_input.strip()
if not stripped:
    print("Name is empty")  # ✓ Works!
```

## Guided Practice

### Peer Code Review Session (Full Period)

**Phase 1: Exchange Code (5 min)**
- Push your code to a shared folder or email it
- Partner receives your code
- Both have time to read through it

**Phase 2: Ask Questions (10 min)**
- Review partner's code
- Write down questions and observations
- Don't assume — ask for clarification

**Phase 3: Share Feedback (10 min)**
- Tell partner what you found
- Use the feedback checklist
- Be constructive, not critical

**Phase 4: Receive Feedback (10 min)**
- Listen to what your partner found
- Ask clarifying questions
- Don't defend — just listen

**Phase 5: Plan Improvements (10 min)**
- What feedback will you implement?
- What's lower priority?
- Make a list for Day 160

**Phase 6: Collaborate on Hard Bugs (10 min)**
- If either of you has a stubborn bug, debug together
- One person codes, other suggests
- Use print statements to trace execution

## Practice Problems

**Problem 1 — Code Review Feedback:** Your peer wrote this function:
```python
def f(x):
    t = 0
    for i in x:
        t += i
    return t / len(x)
```
What feedback would you give?

**Problem 2 — Bug Collaboration:** You both find this bug:
```python
scores = [85, 92, 78]
total = 0
for score in scores:
    total = score  # Bug!
average = total / len(scores)
print(average)  # Prints 78.0, should be 85.0
```
How do you fix it together?

**Problem 3 — Self-Review:** Review your own code. What would a peer notice that you might have missed?

<details>
<summary>💡 Hints</summary>

- Problem 1: Unclear variable names (f, x, t), no docstring, no comments
- Problem 2: total is being overwritten, not accumulated. Change `total = score` to `total += score`
- Problem 3: Check: Are names clear? Is output formatted nicely? Are edge cases handled?
