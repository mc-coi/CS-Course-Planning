# Day 163: Input Validation and Error Handling

**Learning Target:** I can anticipate user errors and handle them gracefully without crashes.

---

## Key Concepts

### Why Error Handling Matters

A program that crashes on bad input is **not professional**. Users will:
- Think your program is broken
- Lose trust in it
- Stop using it

A program that handles errors gracefully:
- Guides users to correct input
- Never crashes
- Builds trust

### Error Handling Patterns

**Pattern 1: Validate and Retry**
```python
while True:
    try:
        score = float(input("Score (0-100): "))
        if 0 <= score <= 100:
            return score
        else:
            print("Score must be between 0 and 100.")
    except ValueError:
        print("Please enter a number.")
```

**Pattern 2: Provide Default**
```python
try:
    max_students = int(input("Max students (default 30): "))
except ValueError:
    max_students = 30
```

**Pattern 3: Try Again with Limit**
```python
attempts = 0
while attempts < 3:
    try:
        score = float(input("Score: "))
        return score
    except ValueError:
        attempts += 1
        if attempts < 3:
            print(f"Try again ({3-attempts} attempts left)")
        else:
            print("Too many invalid attempts. Using 0.")
            return 0.0
```

## Common Error Handling Cases

```python
# Case 1: Empty input
name = input("Name: ").strip()
if not name:
    print("Name cannot be empty")

# Case 2: Invalid type
try:
    age = int(input("Age: "))
except ValueError:
    print("Age must be a number")

# Case 3: Out of range
score = float(input("Score: "))
if not (0 <= score <= 100):
    print("Score must be 0-100")

# Case 4: File operations
try:
    with open("data.txt") as f:
        data = f.read()
except FileNotFoundError:
    print("File not found. Using default data.")
    data = ""

# Case 5: List access
if index < len(list):
    item = list[index]
else:
    print("Index out of range")

# Case 6: Division by zero
if len(scores) > 0:
    avg = sum(scores) / len(scores)
else:
    print("Cannot average empty list")
```

**Step 1: Identify Risky Operations (10 min)**
- Where do users enter data?
- Where do you do division?
- Where do you access lists?
- Where do you do file I/O?

- Wrap in try/except
- Validate inputs
- Provide helpful error messages
- Test with bad input

**Step 3: Test Edge Cases (10 min)**
- Empty input
- Wrong type (letters instead of numbers)
- Out of range
- Files that don't exist

**Step 4: Refine Error Messages (5 min)**
- Are they helpful to users?
- Do they say what to do next?
