# Day 155: Prototype Demo and Feedback

**Learning Target:** I can demonstrate my MVP, gather constructive feedback, and plan improvements based on that feedback.

---

## Key Concepts

### Demo Best Practices

**Before You Demo:**
- [ ] Program runs without crashing (on normal input)
- [ ] You have test inputs written down
- [ ] You know which features work and which are incomplete
- [ ] You can explain the core logic in 1-2 sentences

**During the Demo:**
- Speak clearly
- Use your prepared test inputs
- Show what the program does, not how the code works
- Be honest about what's not yet implemented

**Example Demo Script:**
```
"Here's my Grade Tracker MVP. Right now it lets you enter a
student's name and their test scores, calculates the average
and assigns a letter grade. Watch:

[Enters: Alice, 85, 92, 78, done]

It calculated the average as 85.0 and assigned a B. What's
NOT yet implemented is saving to a file — I'll add that
next week. The error handling is also basic — if you type
invalid data it asks again, but it doesn't have all edge
cases covered.

What do you think? Any suggestions?"
```

### Giving Good Feedback

When reviewing someone else's MVP:

**Do:**
- Try to use the program
- Ask: "What happens if...?"
- Point out what works well
- Ask clarifying questions

**Don't:**
- Criticize the code (it's not polished yet)
- Make demands ("You should...")
- Focus on bugs in the demo

**Good Feedback:**
```
"I like how simple the menu is. One question:
when I enter a score, does it tell me if it was
accepted? And what happens if I try to view
a student that doesn't exist yet?"
```

### Feedback Categories

**What works:** Celebrate what's good
**What's unclear:** "I'm confused about..."
**What's missing:** "It would be nice if..."
**What's confusing:** "This part didn't make sense..."

## Guided Practice

### Demo & Feedback Session (Full Period)

**Phase 1: Prepare Your Demo (10 min)**
- Write down 3 test cases
- Know your demo script (30 seconds)
- Make sure code runs without errors
- Identify 2-3 known limitations

**Phase 2: Demo to Peer (15 min)**
- Explain what your program does (1-2 min)
- Run through test cases (2-3 min)
- Explain features implemented vs. not yet (1 min)
- Pause for questions (1 min)

**Phase 3: Receive Feedback (5 min)**
- Listen without defending
- Write down all feedback
- Ask: "What would be most helpful to improve first?"

**Phase 4: Give Feedback to Partner (15 min)**
- Try their program
- Ask clarifying questions
- Identify what works and what's unclear
- Give constructive suggestions

**Phase 5: Plan Improvements (10 min)**
- Review the feedback you received
- Mark feedback as:
  - Bug fix needed
  - Nice improvement
  - Can defer to later
- Plan which feedback to address in Sprint 1

## Practice Problems

**Problem 1 — Feedback Evaluation:** You receive this feedback: "Your app is slow." Is this helpful? How would you respond?

**Problem 2 — Demo Scenario:** Your program crashes if user enters just spaces. How would you demo this gracefully, and what would you say?

**Problem 3 — Plan Improvements:** You got this feedback:
- "Can you add user authentication?" (complex)
- "The menu could be clearer" (quick fix)
- "Add sorting by name" (medium)

Which will you do first? Why?

<details>
<summary>💡 Hints</summary>

- Problem 1: "Slow" is vague. Ask: "Which part? Is it the input, calculation, or display?" Then prioritize
- Problem 2: Don't hide it. Say: "I found a bug — spaces-only names crash it. That's on my fix list for next week."
- Problem 3: Menu clarity first (quick win, improves demo), then sorting (adds value), defer authentication (too big for MVP)

</details>
