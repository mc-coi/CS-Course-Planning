# Unit 7 - Project Development: Pacing & Differentiation Guide

## Unit Overview
- **Total days allocated:** 35 days / 7 weeks (Days 146–180, 55-minute periods)
- **Core topics covered:** Software Development Lifecycle (SDLC), project planning and proposals, pseudocode and flowcharts, breaking projects into milestones, user interface design, data structure design, function design, MVP (minimum viable product), iterative development, code organization and structure, testing strategies, debugging, refactoring, documentation, user guides, technical guides, presentations, reflection and growth
- **Prerequisites students need:** Mastery of Units 1–6 (all language features, debugging, documentation); ability to independently plan and build programs; collaboration and communication skills

---

## Unit 7 Structure (Days 146–180)

This unit is **entirely project-based**. Students plan, design, build, test, document, and present a complete Python application. The unit follows the Software Development Lifecycle (SDLC):

**Phase 1: Define & Plan (Days 146–150)** → **Phase 2: Design (Days 151–155)** → **Phase 3: Code Sprint 1 (Days 156–160)** → **Phase 4: Code Sprint 2 (Days 161–165)** → **Phase 5: Polish & Test (Days 166–170)** → **Phase 6: Document & Present (Days 171–180)**

---

### Week 30 (Days 146–150): Define & Plan Your Project
**Focus:** Understanding SDLC, writing project proposals, planning with pseudocode/flowcharts, breaking projects into milestones

#### Essential days (cannot skip):
- **Day 146:** Software Development Lifecycle overview — frames the entire unit
- **Day 147:** Writing a project proposal (problem statement, audience, features)
- **Day 148:** Planning with pseudocode and flowcharts
- **Day 149:** Breaking projects into milestones and tasks
- **Day 150:** Peer review of proposals and feedback

#### Flexible days (can compress or cut if pressed for time):
- None. Planning is critical for success; skipping means projects will be poorly organized.

#### If ahead of pace:
- Have students plan more complex features (multiple users, persistence, complexity)
- Challenge them to create detailed technical specifications
- Have them plan for edge cases and error handling upfront

#### If behind pace:
- Provide project proposal template (fill-in-the-blank)
- Simplify project scope or provide project ideas
- Extend planning with guided scaffolding
- **Minimum viable coverage:** Days 147–148 (proposal, planning)

---

### Week 31 (Days 151–155): Design Your Project
**Focus:** Designing UI, data structures, and function signatures; building a working MVP (minimum viable product)

#### Essential days (cannot skip):
- **Day 151:** User interface design — sketching menus, interaction flows
- **Day 152:** Data design — planning data structures (variables, lists, dictionaries)
- **Day 153:** Function design — planning function signatures and their purposes
- **Day 154:** Building a working prototype (MVP) — core functionality only, no polish
- **Day 155:** MVP demo and peer feedback

#### Flexible days (can compress or cut if pressed for time):
- None. Design prevents wasted coding effort.

#### If ahead of pace:
- Have them design for multiple user roles or complex workflows
- Challenge them to design for scalability (what if data grows 10x?)

#### If behind pace:
- Provide design templates (UI mockups, data structure diagrams, function list)
- Simplify scope or provide partially designed projects
- Allow pair design
- **Minimum viable coverage:** Days 153–154 (function design, MVP)

---

### Week 32 (Days 156–160): Core Development Sprint 1
**Focus:** Implementing core features with quality; systematic development and testing

#### Essential days (cannot skip):
- **Day 156:** Setting up project structure (main() function, helper functions, code organization)
- **Days 157–158:** Core feature development — work days with check-ins
- **Day 159:** Code review and collaborative debugging — peer feedback
- **Day 160:** Sprint retrospective — what worked? what needs adjustment?

#### Flexible days (can compress or cut if pressed for time):
- None. These are hands-on work days.

#### If ahead of pace:
- Have students complete more features or higher-quality implementations
- Challenge them to optimize for performance or readability

#### If behind pace:
- Extend work days (reduce other activities)
- Provide code templates or starter code
- Pair weaker students with stronger mentors
- **Minimum viable coverage:** Days 157–160 (development, review, retrospective)

---

### Week 33 (Days 161–165): Core Development Sprint 2
**Focus:** Adding advanced features, input validation, and file I/O; ensuring robustness

#### Essential days (cannot skip):
- **Days 161–162:** Additional features development — work days with check-ins
- **Day 163:** Input validation and error handling — making code robust
- **Day 164:** File I/O integration — saving/loading data for persistence
- **Day 165:** Sprint retrospective — assessing progress

#### Flexible days (can compress or cut if pressed for time):
- None. These are work days.

#### If ahead of pace:
- Have students implement advanced features or user customization
- Challenge them to handle complex data scenarios

#### If behind pace:
- Simplify additional features; focus on core functionality + one extra feature
- Defer file I/O if time is too tight; can be added in Week 34
- Allow pair programming
- **Minimum viable coverage:** Days 161–163 (features, validation)

---

### Week 34 (Days 166–170): Polish & Test
**Focus:** Ensuring quality, reliability, and user experience through systematic testing and refinement

#### Essential days (cannot skip):
- **Day 166:** Code cleanup — comments, docstrings, variable names, organization
- **Day 167:** Testing — comprehensive testing with edge cases and stress testing
- **Day 168:** Bug fixing day — addressing issues found in testing
- **Day 169:** User experience polish — menus, output formatting, clarity
- **Day 170:** Feature freeze — no new features; final code review only

#### Flexible days (can compress or cut if pressed for time):
- None. Quality matters; don't skip polish.

#### If ahead of pace:
- Have them implement advanced testing or optimization
- Challenge them to improve code style and readability beyond requirements

#### If behind pace:
- Reduce scope of testing (focus on main features)
- Simplify polish requirements (basic formatting sufficient)
- **Minimum viable coverage:** Days 167–169 (testing, bug fixes, polish)

---

### Week 35 (Days 171–175): Documentation & Presentation Prep
**Focus:** Preparing to share your work with clear documentation and confident presentation

#### Essential days (cannot skip):
- **Day 171:** Writing documentation (README, user guide, technical guide)
- **Day 172:** Code walkthrough practice — explaining code clearly
- **Day 173:** Building a presentation — slides, demo script, narrative
- **Day 174:** Presentation rehearsal — timing, feedback, refinement
- **Day 175:** Final project submission

#### Flexible days (can compress or cut if pressed for time):
- None. Documentation and presentation are critical final steps.

#### If ahead of pace:
- Have them create video documentation or interactive presentations
- Challenge them to create advanced user guides with screenshots

#### If behind pace:
- Provide documentation templates
- Simplify presentation requirements
- Allow pair presentations
- **Minimum viable coverage:** Days 171–173 (documentation, presentation prep)

---

### Week 36 (Days 176–180): Presentations & Course Wrap-Up
**Focus:** Sharing achievements, reflecting on growth, celebrating learning

#### Essential days (cannot skip):
- **Days 176–177:** Project presentations — students demo and explain projects
- **Day 178:** Course reflection — looking back on growth
- **Days 179–180:** Course celebration, portfolio review, and looking ahead

## Differentiation Strategies

### For Struggling Students
**Scaffolding throughout the unit:**
- **Project selection:** Provide a list of pre-designed projects to choose from (rather than blank slate)
- **Project complexity:** Reduce scope — simple feature set, fewer functions, smaller data structures
- **Planning support:** Provide templates (proposal template, design checklist, milestone checklist)
- **Peer support:** Pair with a stronger student; regular check-ins with teacher
- **Milestone checkpoints:** Break work into smaller, achievable pieces; celebrate progress

**Suggested pacing adjustments:**
- Spend extra time in Weeks 30–31 (planning/design); less time in later weeks
- Provide code templates or starter code
- Focus on MVP only; defer nice-to-have features
- Allow more check-in time during sprint weeks

**Project ideas for struggling students:**
- Grade calculator (input scores, display average, letter grade)
- Simple quiz game (questions, scoring, results)
- To-do list manager (add, remove, display tasks; save to file)
- Number guessing game with hints
- Simple inventory tracker (add items, remove, display)

---

### For On-Track Students
**Standard project expectations:**
- **Scope:** 5–7 core features; well-planned and organized
- **Code quality:** Clean, well-documented, proper error handling
- **Testing:** Edge cases identified and handled
- **Time management:** Meet all deadlines; complete all phases

**Project ideas for on-track students:**
- Grade analyzer (import scores, calculate statistics, identify struggling students)
- Mini text adventure game (rooms, inventory, choices)
- Student record system (add, edit, delete, search students; save to file)
- Library management system (books, borrowing, return tracking)
- Simple expense tracker (categories, budgets, reporting)

---

### For Advanced Students
**Extension activities & expectations:**
- **Scope:** 8+ features; sophisticated design; advanced functionality
- **Code quality:** Modular, well-designed, optimized
- **Features:** File I/O, advanced error handling, data persistence, user customization
- **Design patterns:** Consider design patterns (MVC, observer, factory, etc.)
- **Optimization:** Performance, memory usage, code elegance

**Project ideas for advanced students:**
- Email/messaging application (compose, send, read messages; save to file)
- Chat system with multiple users and data persistence
- Data analysis tool (import CSV, filter, analyze, generate reports)
- Text-based RPG with inventory, combat, and progression
- Collaborative list/task application with save/load and user profiles
- Password-protected journal or note-taking application
- Mini spreadsheet application with formulas and data analysis

## Suggested Checkpoints & Deliverables

### Day 150: Project Proposal
Deliverable: Written proposal including problem statement, audience, core features, nice-to-have features, inputs, outputs. Peer feedback provided.

### Day 155: MVP Demo
Deliverable: Working prototype demonstrating core functionality. Present to class. Receive peer feedback on user experience.

### Day 160: Sprint 1 Retrospective
Deliverable: Brief reflection on progress: What worked? What needs adjustment? What will you focus on next?

### Day 165: Sprint 2 Retrospective
Deliverable: Assessment of additional features. Are they working? What testing revealed?

### Day 170: Feature Freeze Checklist
Deliverable: Code review checklist: Is code clean? Are edge cases handled? Is testing complete?

### Day 175: Final Submission
Deliverable: Complete project code, documentation, and presentation materials.

### Days 176–177: Final Presentation
Deliverable: 5–7 minute presentation demonstrating project, explaining design choices, reflecting on learning.

## Common Struggles & Solutions

### Time Management
**Problem:** Students run out of time or procrastinate
**Solution:**
- Weekly milestones with teacher check-ins
- Public progress tracking (what's done? what's next?)
- Reduced scope for students who fall behind

### Scope Creep
**Problem:** Students add features beyond plan; never finish
**Solution:**
- Feature list locked after Day 150; Day 170 feature freeze strictly enforced
- "Nice to have" list for post-completion only
- Teacher discourages mid-project scope changes

### Debugging Complex Code
**Problem:** Large projects are hard to debug
**Solution:**
- Encourage incremental development (code one feature, test it, move on)
- Pair debugging during Week 32–33
- Systematic debugging strategies (print statements, trace execution)

### Documentation Paralysis
**Problem:** Students skip documentation or write too much
**Solution:**
- Provide documentation templates with required sections
- Focus on clarity: What does the program do? How do you use it? How does it work?
- Review documentation drafts during Week 35

### Presentation Anxiety
**Problem:** Students are nervous about presenting
**Solution:**
- Practice presentations in Week 35
- Focus on explaining your code (what does it do?) not defending it
- Peer feedback normalizes presentations

## Notes on Unit 7 Assessment

Unit 7 is **assessed through the final project**, not a traditional test. Assessment includes:

1. **Project Quality (50%)**
   - Meets specifications and requirements
   - Code is clean, well-organized, properly documented
   - All planned features work; error handling in place
   - Testing is thorough (edge cases considered)

2. **Process & Collaboration (25%)**
   - Planning and design are thoughtful
   - Regular progress; meets milestones
   - Peer feedback incorporated
   - Documentation is clear and complete

3. **Presentation & Communication (15%)**
   - Presentation clearly explains project
   - Code walkthrough demonstrates understanding
   - Reflection shows learning and growth
   - Professional communication

4. **Reflection (10%)**
   - Student can articulate what they learned
   - Recognizes strengths and growth areas
   - Identifies next steps or future improvements

---

## Grade Distribution Guidance

| Criterion | Excellent | Good | Satisfactory | Needs Work |
|-----------|-----------|------|--------------|-----------|
| **Functionality** | All features work; robust error handling | Most features work; some edge cases missed | Basic features work; minimal error handling | Features incomplete or non-functional |
| **Code Quality** | Clean, modular, well-documented | Generally clean; mostly well-organized | Adequate organization; minimal documentation | Disorganized; hard to follow |
| **Design & Planning** | Thoughtful design; clear architecture | Good design; some optimization missed | Basic design; some unclear choices | Minimal planning evident |
| **Testing** | Comprehensive testing; edge cases found/fixed | Good testing; most issues caught | Basic testing; some issues remain | Little testing; bugs present |
| **Presentation** | Clear, professional, well-rehearsed | Good explanation; minor clarifications needed | Adequate; some confusion | Unclear or unprepared |

## Reflection Prompts for Students

At the end of Unit 7 (and Coding I), students should reflect:

1. **What was the most challenging part of your project? How did you overcome it?**
2. **What design decision are you most proud of? Why?**
3. **If you could start over, what would you do differently?**
4. **Which programming concept do you feel most confident in? Which needs more work?**
5. **How has your problem-solving approach evolved since Day 1?**
6. **What's one thing you built this year that surprised you?**
7. **What would you want to learn next in Coding II?**

## Looking Ahead: Coding II Prerequisites

Students completing Unit 7 should be ready for Coding II topics:
- Object-Oriented Programming (classes, objects, inheritance)
- Advanced data structures (dictionaries, sets, tuples)
- Working with external libraries (requests, pandas, matplotlib)
- Databases and SQL

- Advanced algorithms and data structures
- Collaborative development (git, GitHub)

Encourage students to explore these areas during the summer or in independent study!

- Plan and execute a moderately complex software project independently

- Document thoroughly for user and developer audiences

- Reflect on their learning and growth
