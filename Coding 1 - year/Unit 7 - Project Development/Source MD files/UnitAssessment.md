# Unit 7 Assessment: Project Rubric and Self-Evaluation

## Overview

Your capstone project is assessed on **100 points** across five categories:

| Category | Points | Focus |
|----------|--------|-------|
| **Functionality** | 20 | Does the program work? |
| **Code Quality** | 20 | Is the code well-written? |
| **Testing & Robustness** | 20 | Does it handle edge cases and errors? |
| **Documentation** | 20 | Is it explained and documented? |
| **Presentation** | 20 | Can you present and explain it? |
| **TOTAL** | **100** | |

---

## Detailed Rubric

### 1. Functionality (20 points)

**What This Measures:** Does your program do what it's supposed to do?

#### 18–20 points: Excellent
- [✓] All core features work perfectly
- [✓] Additional features work well
- [✓] Program behaves as specified in proposal
- [✓] No unexpected behavior
- [✓] All user interactions work correctly

#### 14–17 points: Good
- [✓] All core features work
- [✓] Most features work perfectly
- [✓] Minor issues that don't prevent use
- [✓] Overall meets proposal specifications

#### 10–13 points: Satisfactory
- [✓] Most core features work
- [✓] Some features incomplete or buggy
- [✓] Program still usable despite issues
- [✓] Partially meets proposal

#### 6–9 points: Developing
- [✓] Some core features work
- [✓] Multiple features incomplete or broken
- [✓] Program has significant issues

#### 0–5 points: Needs Improvement
- [ ] Core features missing or broken
- [ ] Program doesn't run
- [ ] Doesn't meet proposal specifications

---

### 2. Code Quality (20 points)

**What This Measures:** Is the code well-organized and professional?

#### 18–20 points: Excellent
- [✓] Code is well-organized with clear sections
- [✓] All functions are short and focused (~15-20 lines max)
- [✓] All functions have descriptive names (verb-noun: `calculate_average`)
- [✓] All functions have complete docstrings
- [✓] Variables have clear, descriptive names
- [✓] No code duplication (DRY principle)
- [✓] Consistent indentation and style

#### 14–17 points: Good
- [✓] Code is organized and readable
- [✓] Most functions are appropriately sized
- [✓] Most function/variable names are descriptive
- [✓] Most functions have docstrings
- [✓] Minor style inconsistencies

#### 10–13 points: Satisfactory
- [✓] Code is somewhat organized
- [✓] Some functions are too long or unclear
- [✓] Some function/variable names unclear
- [✓] Some missing docstrings
- [✓] Some code duplication

#### 6–9 points: Developing
- [✓] Code is disorganized
- [✓] Many functions are too long or unclear
- [✓] Poor naming choices
- [✓] Few or no docstrings
- [✓] Significant code duplication

#### 0–5 points: Needs Improvement
- [ ] Code is hard to understand
- [ ] Poor organization and naming
- [ ] No documentation

---

### 3. Testing & Robustness (20 points)

**What This Measures:** Does the program handle edge cases and errors without crashing?

#### 18–20 points: Excellent
- [✓] Handles all normal cases perfectly
- [✓] Tested with edge cases (empty, 0, max values)
- [✓] Handles error cases gracefully (bad input, missing file)
- [✓] Program never crashes unexpectedly
- [✓] Error messages are helpful to users
- [✓] Validates all user input
- [✓] File operations protected with try/except

#### 14–17 points: Good
- [✓] Handles most cases correctly
- [✓] Tested with some edge cases
- [✓] Handles most error cases
- [✓] Few unexpected crashes
- [✓] Most input validated
- [✓] File operations mostly protected

#### 10–13 points: Satisfactory
- [✓] Handles normal cases
- [✓] Some edge cases tested
- [✓] Some error handling
- [✓] Some crashes on unusual input
- [✓] Inconsistent input validation

#### 6–9 points: Developing
- [✓] Handles some normal cases
- [✓] Little edge case testing
- [✓] Limited error handling
- [✓] Crashes on bad input
- [✓] Missing input validation

#### 0–5 points: Needs Improvement
- [ ] Program crashes frequently
- [ ] No error handling
- [ ] No input validation

---

### 4. Documentation (20 points)

**What This Measures:** Is your work explained and documented?

#### 18–20 points: Excellent
- [✓] Complete README file
- [✓] Clear user guide with examples
- [✓] All functions have docstrings
- [✓] Comments explain complex logic
- [✓] Comments explain "why," not just "what"
- [✓] No dead code (commented-out lines)
- [✓] Module docstring at top of file
- [✓] Clear variable/function naming documents code

#### 14–17 points: Good
- [✓] README file exists and is clear
- [✓] Most functions documented
- [✓] Important logic commented
- [✓] Few issues with documentation

#### 10–13 points: Satisfactory
- [✓] Basic README or documentation
- [✓] Some functions documented
- [✓] Some comments on logic
- [✓] Partially explains program

#### 6–9 points: Developing
- [✓] Minimal documentation
- [✓] Few docstrings
- [✓] Little explanation of logic
- [✓] Hard to understand code

#### 0–5 points: Needs Improvement
- [ ] No documentation
- [ ] No docstrings
- [ ] Unexplained code

---

### 5. Presentation (20 points)

**What This Measures:** Can you present your work effectively?

#### 18–20 points: Excellent
- [✓] Introduction clearly states problem and solution
- [✓] Demo shows features working
- [✓] Explains 1-2 key functions clearly
- [✓] Discusses challenges and solutions
- [✓] Speaks clearly, makes eye contact
- [✓] Paces appropriately (not too fast/slow)
- [✓] Ready for questions
- [✓] Owns the work (no apologies for bugs)

#### 14–17 points: Good
- [✓] Clear introduction and demo
- [✓] Explains code reasonably well
- [✓] Discusses some challenges
- [✓] Generally clear delivery
- [✓] Mostly ready for questions

#### 10–13 points: Satisfactory
- [✓] Demonstrates the program
- [✓] Explains some code
- [✓] Delivery somewhat unclear
- [✓] Partially ready for questions

#### 6–9 points: Developing
- [✓] Presentation is unclear
- [✓] Limited code explanation
- [✓] Delivery is hard to follow
- [✓] Not ready for questions

#### 0–5 points: Needs Improvement
- [ ] Presentation is confusing
- [ ] No clear explanation
- [ ] Delivery is poor

---

## Grade Appeals & Revisions

**If you disagree with your grade:**

1. Review the rubric carefully
2. Identify specific areas where you believe your work exceeds the rubric
3. Request a meeting with your teacher to discuss
4. Be respectful and specific (not defensive)

**Revision Policy:**
- Can fix bugs and resubmit for up to 10 points back
- Must be within 1 week of receiving feedback
- Only major bugs (not style changes)

---

## Next Steps After Assessment

**Regardless of grade:**
1. Save all your work (you'll be proud of it later!)
2. Add to your portfolio
3. Learn from feedback
4. Apply lessons to Coding II projects
5. Keep coding!
