## Presentation Formats

Choose what works best for you:

### Format 1: Live Demo (Recommended)
- Run your app on the projector
- Walk through features as you use them
- More engaging, shows it really works
- Risk: something might crash (have backup plan)

**Structure:**
1. Intro (30 sec): "I built X. Here's what it does."
2. Demo (4–5 min): Show key features
3. Architecture (1 min): Briefly explain code structure
4. Q&A (remaining time): Answer questions

### Format 2: Slides + Code Walkthrough
- Use slides to introduce project, show screenshots
- Walk through key code sections on screen
- Less risk of crashes; more controlled
- Less engaging but professional

**Structure:**
1. Title slide (project name, your name)
2. Overview slide (what does it do, why you built it)
3. Features slide (list of features with screenshots)
4. Code walkthrough (show 2–3 key code examples)
5. Results slide (final thoughts)
6. Q&A

## Demo Script Refinement

Timing is important. Practice and aim for 5–7 minutes:

```
DEMO TIMELINE (Student System)

[0:00] Intro
  "I built a student information system. It lets you manage student
   records with adding, searching, sorting, and saving to JSON."
  [5-10 seconds]

[0:10] Show structure
  "The code uses OOP with Student and StudentSystem classes, plus
   file I/O for persistence."
  [10-15 seconds]

[0:25] Run app
  "Let me show you how it works."
  [Run: python main.py]
  [5 seconds]

[0:35] Demo: Add students
  "I'll add a few students."
  [Pick: 1, add Alice]
  [Pick: 1, add Bob]
  [Pick: 1, add Carol]
  [1 minute]

[1:35] Demo: View all
  "Now let me show all students."
  [Pick: 2]
  [30 seconds]

[2:05] Demo: Search
  "I can search for a student by name."
  [Pick: 3, search "alice"]
  [Shows partial match, case-insensitive]
  [45 seconds]

[2:50] Demo: Sort
  "And sort by GPA."
  [Pick: 4]
  [30 seconds]

[3:20] Demo: Save & restart
  "When I exit, it saves to JSON."
  [Pick: 5]
  [Exit, show students.json created]
  [Restart app, students load automatically]
  [1 minute]

[4:20] Architecture highlight
  "The code is organized with Student and StudentSystem classes.

  [Show code briefly]
  [30 seconds]

[4:50] Summary
  "That's the student system. It's a complete CRUD app with OOP
   design and persistent storage. Questions?"
  [10 seconds]

[5:00] Q&A
  [Up to 7 minutes for questions]
```

## Common Presentation Mistakes to Avoid

- **Too fast** — Speak slowly, pause, let people absorb
- **Mumbling** — Articulate, project your voice
- **Not explaining** — Don't just show features; explain why
- **Assuming knowledge** — Explain what "CRUD" means, not just say it
- **Ignoring questions** — Even if you don't know the answer, say so honestly
- **Defensive** — If someone finds a bug, say "Good catch!"

---

## Backup Plans

If your demo crashes:

**Plan A: Restart**
- Restart the app and pick a different feature to demo

**Plan B: Show the code**
- Pull up your code on screen
- Walk through how a feature works
- "The feature works, but let me show you the code instead"

**Plan C: Use screenshots**
- Take screenshots beforehand of app running
- Show them on slide: "Here's what it looks like when you run it"

---

## Presenter Confidence Tips

- **Practice out loud** — Don't just think through it; actually say it
- **Time yourself** — Use a timer; aim for 5–7 minutes
- **Anticipate questions** — What might someone ask? Prep an answer
- **Know your code** — Review the code you'll show on screen
- **Take a deep breath** — Before you present, breathe
- **Remember:** You're an expert on YOUR code. Own it.
