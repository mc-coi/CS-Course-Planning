# Unit 7 - Capstone Project: Pacing & Differentiation Guide

## Unit Overview

- **Total days allocated:** 25 days / 5 weeks
- **Core focus:** Real-world application development combining all course concepts
- **Big Idea:** From planning through presentation, students build a complete working project
- **Key Topics Covered:**
  - Project planning and requirements analysis
  - Object-oriented design and architecture
  - File I/O and data persistence
  - Integration of APIs or external libraries
  - Algorithm design (sorting, searching, recursion where applicable)
  - Testing and debugging
  - Code documentation and presentation
  - Real-world development workflow
- **Prerequisites students need:**
  - All Units 1–6 concepts mastered
  - Ability to work independently with guidance
  - Problem-solving skills for novel scenarios

---

## Unit Structure & Pacing

### Phase 1: Planning & Design (Days 1–5)

#### Days 1–2: Project Selection & Requirements Gathering
- **Essential days** (must teach): YES
  - Students must understand project scope and requirements before coding
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Have students refine requirements, add stretch goals
- **If behind pace:** Provide pre-defined project options with detailed requirements

Students choose from 4 capstone project options:
1. **Data Analyzer** — Read CSV/JSON, compute statistics, generate visualizations
2. **API-Powered App** — Fetch live data from public APIs, display and analyze results
3. **Student Information System** — CRUD application with file persistence
4. **Custom Project** — Student-proposed project (with teacher approval)

**Key Activities:**
- Read project descriptions
- Define core features (must-haves)
- Identify stretch goals (nice-to-haves)
- Write simple requirements document

#### Days 3–5: Object-Oriented Design & Architecture Planning
- **Essential days** (must teach): YES
  - Proper design before coding saves time and produces better code
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Complex class hierarchies, design patterns, detailed UML diagrams
- **If behind pace:** Simple class lists, basic data flow diagrams, templates

**Key Activities:**
- Design class structure (class names, attributes, methods)
- Plan data persistence strategy
- Create pseudo-code for major functions
- Design user interface (command-line menus, output format)
- Identify which Unit 1–6 concepts will be used

**Deliverables:**
- Class diagram or written class specifications
- Data structure design (how data will be stored)
- Pseudo-code for key algorithms
- Brief design document (1-2 pages)

### Phase 2: Implementation (Days 6–19)

#### Days 6–10: Core Functionality Development
- **Essential days** (must teach): YES
  - First fully working version of main features
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Add advanced features, optimization, error handling
- **If behind pace:** Focus on must-have features only; skip stretch goals

**Key Activities:**
- Create class definitions and __init__ methods
- Implement core methods for each class
- Build file I/O (reading/writing data)
- Implement basic data processing logic
- Test basic functionality

**Milestone:** End of Day 10, students have a working prototype with core features

#### Days 11–14: Data Processing & Advanced Features
- **Essential days** (must teach): YES
  - Adding the sophistication that makes projects impressive
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Multiple data sources, complex algorithms, optimization
- **If behind pace:** Focus on single data source, basic processing

**Key Activities:**
- Implement sorting/searching (Unit 5 concepts)
- Add data analysis and reporting
- Integrate APIs if applicable (Unit 6)
- Implement statistics or calculations
- Add user menu system and navigation

**Milestone:** End of Day 14, project has all major features working

#### Days 15–17: Error Handling, Testing & Debugging
- **Essential days** (must teach): YES
  - Production code requires robust error handling and testing
- **Flexible days** (can compress or cut if pressed for time): Can be compressed but NOT skipped
- **If ahead of pace:** Automated testing, edge case handling, performance optimization
- **If behind pace:** Basic try/except for common errors, manual testing

**Key Activities:**
- Add exception handling (Unit 4)
- Validate user input
- Test edge cases and error scenarios
- Fix bugs discovered during testing
- Test integration of all components

**Milestone:** End of Day 17, project is stable and handles errors gracefully

#### Days 18–19: Polish & Optimization
- **Essential days** (must teach): YES
  - Making code production-ready
- **Flexible days** (can compress or cut if pressed for time): Can compress to 1 day
- **If ahead of pace:** Performance optimization, memory efficiency, advanced features
- **If behind pace:** Code cleanup, adding comments, user interface polish

**Key Activities:**
- Refactor for readability and efficiency
- Add helpful comments and docstrings
- Improve user interface and output formatting
- Add optional advanced features if time permits
- Final testing and bug fixes

**Milestone:** End of Day 19, project is production-ready

### Phase 3: Documentation & Presentation (Days 20–25)

#### Days 20–21: Code Documentation & README
- **Essential days** (must teach): YES
  - Code that isn't documented is hard to maintain or reuse
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Comprehensive documentation, API documentation, architecture guide
- **If behind pace**: Basic README and docstrings only

**Key Activities:**
- Write docstrings for all classes and methods
- Create README.md with project description and usage instructions
- Document any external dependencies or setup requirements
- Add code comments for complex logic
- Create simple user guide if applicable

**Deliverables:**
- README.md (project description, how to run, features)
- Docstrings in all code
- Inline comments for complex sections

#### Days 22–23: Presentation Preparation & Practice
- **Essential days** (must teach): YES
  - Ability to communicate about code is as important as the code itself
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Live demo, fielding advanced questions, showing code design choices
- **If behind pace:** Structured presentation outline, speaking points provided

**Key Activities:**
- Create presentation slides (3-5 slides recommended)
- Prepare live demo of project
- Write speaking points
- Practice presentation and time it
- Prepare for common questions

**Presentation Should Cover:**
- What the project does (elevator pitch)
- Design decisions and architecture
- Key technical challenges and solutions
- Live demo of core features
- What was learned

**Presentation Duration:** 5-10 minutes (adjust based on class size)

#### Days 24–25: Project Presentations & Assessment
- **Essential days** (must teach): YES
  - Presentations are the capstone; all students must present
- **Flexible days** (can compress or cut if pressed for time): NONE
- **If ahead of pace:** Detailed Q&A, peer review, extended presentations
- **If behind pace:** Simplified presentations, written summary alternative available

**Day 24:** Class presentations (½ to full class period depending on class size)
**Day 25:** Final feedback, reflection, assessment completion

## Differentiation Strategies

### For Struggling Students

**Project recommendations:**
- **Student Information System** — Clear CRUD structure, familiar domain
- Avoid API projects unless student has strong interest
- Avoid complex data analysis projects

**Specific scaffolding:**
- Provide **project templates** with class stubs and method signatures
- Create **detailed requirements checklist** with exact features to implement
- Provide **code examples** from previous units that apply to their project
- Offer **daily check-ins** to ensure progress and address blockers
- Allow **simplified scope** — focus on core features, skip stretch goals
- Provide **error handling templates** for common scenarios
- Provide **presentation outline** and talking points

**Pacing adjustments for struggling students:**
- Spend 2 days on Days 1–2 (planning) with more guidance
- Provide more scaffolding during Days 6–14 (implementation)
- Simplify stretch goals; make core features very achievable
- Allow Day 18–19 to focus on getting features working rather than optimization
- Provide presentation script/outline rather than expecting extemporaneous presentation
- Consider alternative assessment: code review + written project summary instead of presentation

**Support strategies:**
- Pair with stronger students for brainstorming
- Daily 5-minute check-ins to prevent them getting stuck
- Provide working code examples for reference
- Focus on core features; celebrate achieving the MVP
- Help them explain the "why" behind their design choices

### For On-Track Students

**Project recommendations:**
- Any of the four capstone options
- Can choose a custom project with clear scope

**Standard path:**
- Follow the 5-phase breakdown as outlined
- Implement all core features fully
- Complete meaningful error handling and testing
- Create professional documentation
- Give a 5-minute presentation

**What to expect by milestone:**
- **End of Day 5:** Clear design with class diagrams and pseudo-code
- **End of Day 10:** Working core features
- **End of Day 14:** All core features with basic testing
- **End of Day 19:** Production-ready code, well-documented
- **Days 24–25:** Professional presentation

**Pacing tips:**
- Days 1–5 planning should feel deliberate but not slow
- Days 6–14 implementation will be the bulk of work; pace yourself
- Days 15–17 testing is critical; don't rush
- Days 18–19 are satisfying (seeing it work polished)
- Days 20–23 documentation and presentation prep
- Days 24–25 celebration and reflection

### For Advanced Students

**Project recommendations:**
- **Custom project** with ambitious scope
- **API-Powered App** with multiple data sources and visualization
- **Data Analyzer** with statistical analysis and advanced visualizations
- Combine elements from multiple project options

**Extension ideas:**
- Web interface (Flask or Django) instead of command-line
- Database backend instead of JSON/CSV files
- Machine learning integration (prediction, classification)
- Performance optimization and benchmarking
- Advanced visualization and interactive dashboards
- Mobile app interface
- API endpoint creation (building the API they would consume)

**Advanced features to implement:**
- Multi-user system with authentication
- Caching and performance optimization
- Advanced error recovery and logging
- Configuration files for customization
- Automated testing (unit tests, integration tests)
- Continuous integration setup

**Presentation opportunities:**
- Extended presentation (10+ minutes)
- Live Q&A with technical depth
- Code walkthrough of complex sections
- Performance analysis and benchmarking
- Design pattern usage and justification
- Future roadmap and scalability discussion

## Assessment & Grading

### Grading Components (suggested):

1. **Code Implementation (40%)**
   - Core features functional and working
   - Code organization and design
   - Error handling and robustness
   - Use of appropriate data structures and algorithms from Units 1–6

2. **Design & Architecture (15%)**
   - Object-oriented design (Unit 2)
   - Thoughtful class structure and relationships
   - Data structure choices (Unit 1)
   - Algorithm choices appropriate to problem

3. **Documentation (15%)**
   - README and usage instructions
   - Code comments and docstrings
   - Clear variable and function names
   - Design decisions explained

4. **Testing & Quality (15%)**
   - Error handling (Unit 4)
   - Edge case handling
   - User experience
   - Code cleanliness

5. **Presentation (15%)**
   - Clear explanation of project and design
   - Live demo of functionality
   - Ability to answer questions
   - Professional communication

### Rubric by Level:

**Proficient (A/B range):**
- Core features fully implemented and working
- Well-organized OOP design
- Thoughtful error handling
- Clear documentation
- Professional presentation

**Developing (B/C range):**
- Core features working but some incomplete
- Basic OOP design with some organization issues
- Basic error handling
- Minimal documentation
- Adequate presentation

**Beginning (C/D range):**
- Core features partially working or incomplete
- Weak OOP design or organization
- Limited error handling
- Little documentation
- Awkward presentation

## Flexibility Notes

### Timeline Adjustments

**If 1 week ahead:**
- Extended planning phase (more design, more options)
- Implement stretch goals
- More thorough testing
- Extended presentations (10+ minutes)
- Start Unit 8 or begin Capstone II

**If 3 days behind:**
- Reduce project scope (fewer features required)
- Provide more templates and scaffolding
- Compress documentation phase
- Simplify presentations

**If 1 week behind:**
- Very simple project scope (Data Analyzer only, with template)
- Provide class stubs and method signatures
- Focus on core features only, no stretch goals
- Written project summary instead of presentation

**If 2 weeks behind:**
- Choose simplest project option (Student System with template)
- 80% of code provided as templates
- Focus on getting something working
- Minimal documentation
- Very brief presentation or recorded video

### Common Student Struggles

1. **Scope creep** (Days 6–14): Project becomes too large
   - *Prevention:* Clear MVP requirements from Day 1
   - *Fix:* Help students prioritize features; defer nice-to-haves

2. **Getting stuck** (Days 6–14): Progress stalls on one feature
   - *Prevention:* Daily check-ins
   - *Fix:* Help debug, maybe provide working code for that part

3. **Last-minute panic** (Days 18–19): Code isn't ready
   - *Prevention:* Realistic checkpoint milestones
   - *Fix:* Focus on getting core features working, not polish

4. **Weak presentation** (Day 24): Can't explain their work
   - *Prevention:* Practice and outline from Day 22
   - *Fix:* Provide script/talking points, allow notes

### Topics That Go Well

- **Students love:** Building something real, seeing it work, showing family/friends
- **Students struggle with:** Planning before coding, error handling, documentation
- **Best use of APIs:** Weather, crypto, news (engaging real data)
- **Best use of File I/O:** Student records, personal tracking apps, data analysis

## Suggested Checkpoints & Milestones

### Checkpoint 1: End of Day 5 (Design)
- **Deliverable:** Class diagram or written class specs, data design, pseudo-code
- **Assessment:** Will this design support the planned features?
- **Action if not on track:** Redesign before coding

### Checkpoint 2: End of Day 10 (Working Core)
- **Deliverable:** Running program with core features functional
- **Assessment:** Does the MVP work?
- **Action if not on track:** Simplify features, get basics working

### Checkpoint 3: End of Day 14 (All Features)
- **Deliverable:** All planned features implemented (may have bugs)
- **Assessment:** Are all features present? Does it mostly work?
- **Action if not on track:** Debug most critical issues, defer polish

### Checkpoint 4: End of Day 17 (Robust & Tested)
- **Deliverable:** Code handles errors gracefully, tested
- **Assessment:** Is code stable? Does it handle edge cases?
- **Action if not on track:** Add try/except, test edge cases

### Checkpoint 5: End of Day 19 (Production Ready)
- **Deliverable:** Clean, documented, optimized code
- **Assessment:** Could you turn this in as homework for Unit 1–6?
- **Action if not on track:** Prioritize; get core features solid

### Checkpoint 6: Day 24–25 (Presentation & Final Assessment)
- **Deliverable:** Working project, documentation, presentation
- **Assessment:** Can you explain what you built and why?

## Key Success Factors

**For students to succeed:**
1. **Clear requirements** from Day 1 (reduce uncertainty)
2. **Regular check-ins** (prevent getting hopelessly stuck)
3. **Realistic scope** (achievable in timeline with quality)
4. **Working intermediate version** by Day 10 (builds confidence)
5. **Daily progress** (avoiding panic at the end)
6. **Presentation practice** (reduces presentation anxiety)
7. **Celebration of achievement** (capstone is a big deal!)

**Teacher's role:**
- Guide design thinking, don't code for them
- Help debug, don't give solutions
- Ensure scope is realistic
- Monitor progress with checkpoints
- Celebrate effort and learning, not just final product

## Notes on Course Prerequisites

This capstone assumes students have strong foundational knowledge across all Units 1–6. Watch for:

- **Unit 1:** Data structure design — are they using appropriate structures?
- **Unit 2:** OOP design — do classes make sense?
- **Unit 3:** Algorithms — are they using appropriate search/sort?
- **Unit 4:** File I/O and error handling — is their code robust?
- **Unit 5:** Algorithm analysis — are their choices efficient?
- **Unit 6:** APIs and libraries — are they using external tools effectively?

If students are weak in specific areas, help them recognize it and work with their strengths.

---

## Time Management Tips

- **Start planning immediately:** Days 1–5 are critical; don't rush
- **Prototype early:** Get something working by Day 10
- **Test continuously:** Don't wait until Day 15
- **Document as you go:** Don't leave documentation to the last minute
- **Practice presentation:** Day 22 isn't too early to start
- **Celebrate progress:** Students have done an amazing job by Day 25

---

## Final Thoughts

The Capstone Project is the culmination of a full year of learning. Students are building real software. This is exciting, challenging, and absolutely achievable with proper guidance. Your role is to:

1. Set clear expectations and realistic scope
2. Provide scaffolding where needed
3. Let students struggle a little (productive struggle builds confidence)
4. Celebrate their work and learning
5. Help them see they're capable of building real things

Students will remember this capstone project for years. Make it count!
