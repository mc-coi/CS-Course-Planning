# Unit 8: Capstone Project

> Build, polish, and present a complete original game — applying everything from Units 1–7 in one ambitious, creative project.

## Unit Overview

- **Duration:** 25 days / 5 weeks
- **Big Idea:** Real game development is iterative — plan, prototype, test, polish, ship
- **TN Standards:** All VGD standards applied; 9-12.VGD.8 — Design and present a complete original game
- **Deliverables:** Playable game, GDD, exported build, presentation

## Learning Targets

- I can write a complete Game Design Document for an original game
- I can build a playable prototype in Unity within one sprint
- I can implement all major game systems: movement, collision, UI, audio, enemies
- I can iterate based on playtest feedback
- I can export a standalone build of my game
- I can present my game and development process professionally

## Project Options

Choose ONE of the following five projects to build over the 25 days:

### Option 1: Platformer Adventure

**Concept:** A 2D side-scrolling platformer where the player jumps, avoids obstacles, and collects items.

**Required Systems:**
- Player movement and jumping (using Rigidbody2D and physics)
- At least 3 different types of platforms or obstacles (moving platforms, spikes, ice)
- At least 2 enemy types with basic AI (patrol and chase)
- Collectibles (coins, health pickups) with feedback
- Health/lives system
- At least 3 levels with increasing difficulty
- Scene transitions and a main menu
- Win and lose conditions

**Design Tips:**
- Focus on tight, responsive controls. Movement is EVERYTHING in platformers.
- Use different colored platforms to teach the player what's dangerous vs. safe.
- Playtesting is crucial. Ask peers: "Is this fun? Is the difficulty fair?"

**Example Mechanics:**
- Double jump
- Wall slide / wall jump
- Dash ability
- Time-based challenges (collect items before timer ends)

---

### Option 2: Top-Down Action Game (Bullet Hell Style)

**Concept:** A top-down arcade game where the player dodges incoming bullets, enemies, or obstacles while trying to survive or defeat enemies.

**Required Systems:**
- Player movement (WASD or arrow keys, smooth and responsive)
- Spawn system for enemies or bullets
- Player health and invulnerability frames (brief period after taking damage)
- At least 2 different enemy types with different attack patterns
- Player attack system (bullets, melee, or AOE)
- Scoring/wave system
- Difficulty increases over time
- Main menu and game over screen

**Design Tips:**
- Space is everything. Make sure the player has room to dodge.
- Vary the difficulty gradually. Early waves should be manageable; later ones punishing.
- Use particle effects to show damage and hits.

**Example Mechanics:**
- Power-ups (rapid fire, shield, extra health)
- Boss fight (one big enemy with multiple attack phases)
- Combo system (hit enemies multiple times for bonus points)

---

### Option 3: Puzzle Game

**Concept:** A logic-based game where the player solves puzzles using spatial reasoning, timing, or pattern recognition.

**Required Systems:**
- Multiple puzzle types (sliding blocks, memory matching, pattern replication, physics-based)
- At least 10 puzzles with increasing difficulty
- Progress tracking (show which puzzles are solved)
- Undo button (for many puzzle games)
- Hint system or tutorial
- Win condition (solve all puzzles or reach a score)
- Main menu and pause functionality

**Design Tips:**
- Make sure each puzzle teaches something. The first puzzle should be trivial.
- Playtest hard. If a puzzle is frustrating, redesign it.
- Visual feedback is crucial. Show what the player is interacting with.

**Example Mechanics:**
- Sliding block puzzle (arrange blocks to match a target)
- Memory game (flip tiles, match pairs)
- Color/pattern matching
- Physics-based logic (ramps, gravity, collision-based triggers)

---

### Option 4: Narrative Adventure / Choice-Based Game

**Concept:** A game where the player makes choices that affect the story. Dialogue trees, branching paths, and multiple endings.

**Required Systems:**
- Multiple scenes (at least 5) with different environments
- Dialogue system with choices (at least 3 branching points)
- Inventory system (optional but good)
- At least 2 different endings based on player choices
- NPCs with interaction prompts
- Quest/objective tracking
- Main menu and pause system

**Design Tips:**
- Write a compelling story first. The rest follows.
- Create a flowchart of your story branches BEFORE coding.
- Use music and visuals to set mood. Narrative games live or die on atmosphere.
- Make choices feel meaningful. If every choice leads to the same outcome, it feels fake.

**Example Mechanics:**
- Dialogue trees
- Inventory-based solutions (use key to open door, give gift to NPC)
- Character relationships (NPCs react differently based on past choices)
- Time-based choices (some choices only available for a short time)

---

### Option 5: Endless Runner / Survival Game

**Concept:** A game where the player survives as long as possible against increasing difficulty. Score is based on survival time or distance traveled.

**Required Systems:**
- Scrolling or continuous movement
- Infinite obstacle generation (or at least enough variety to feel endless)
- Player movement (dodge left/right or jump)
- Health system or one-hit-to-lose mechanic
- Score based on time survived or distance
- High score tracking (PlayerPrefs)
- Difficulty increase over time
- Pause and main menu
- Game over screen with score display

**Design Tips:**
- Balance is tricky. Make sure the difficulty curve is smooth, not sudden spikes.
- Visual feedback is critical. Show the player their score, time, and lives prominently.
- Make obstacles easy to distinguish from the background.

**Example Mechanics:**
- Power-ups (invulnerability, slow-motion, shield)
- Combo system (avoid obstacles in succession for bonus points)
- Multiple characters with different abilities
- Boss encounters every X seconds

## Build & Export Instructions

### Windows Standalone Build

1. **Open Build Settings**
   - File → Build Settings

2. **Add Scenes**
   - Click "Add Open Scenes" to add the current scene
   - Drag any other scenes into the "Scenes In Build" list
   - Order them: MainMenu (0), GameScene(s) (1+), GameOver (last)

3. **Select Platform**
   - In the Platform list, click "PC, Mac & Linux Standalone"
   - Click "Switch Platform" (this may take a minute)

4. **Configure Build Settings**
   - Architecture: x86_64
   - Copy PDB files: Unchecked (unless you need debugging)
   - Development Build: Unchecked (for production)

5. **Build**
   - Click "Build" button
   - Choose a folder (e.g., `Builds/Windows`)
   - Wait for the build to complete
   - You should see a .exe file in the output folder

6. **Test the Build**
   - Run the .exe from outside Unity
   - Play through the entire game
   - Test all buttons, transitions, and features
   - Check that audio and visuals load correctly

**Troubleshooting:**
- If the build crashes, check the console for errors
- Make sure all scripts are error-free before building
- Ensure all audio clips and sprites are imported
- Verify that all scenes are in Build Settings

### WebGL Build

1. **Open Build Settings** (File → Build Settings)

2. **Add Scenes** (same as Windows, above)

3. **Select Platform**
   - Click "WebGL" in the platform list
   - Click "Switch Platform"

4. **Configure WebGL Settings**
   - Compression Format: Brotli (smaller file size)
   - Data Caching: Unchecked (for updates)

5. **Build**
   - Click "Build"
   - Choose output folder (e.g., `Builds/WebGL`)
   - Wait 2–5 minutes (WebGL builds are slow)

6. **Test Locally**
   - You CANNOT just open the HTML file in your browser
   - You need a local web server:
     - **Windows:** Open PowerShell in the build folder, run `python -m http.server 8000`
     - **Mac/Linux:** Open Terminal in the build folder, run `python -m http.server 8000`
   - Open `http://localhost:8000` in your browser
   - Play and test

7. **Deploy (Optional)**
   - Upload the build folder to a web host (Itch.io, GitHub Pages, etc.)
   - Share the link with your class

**Troubleshooting:**
- If WebGL is too large (>100 MB), reduce textures or use compression
- If performance is poor in browser, optimize physics and particles
- Test in both Chrome and Firefox if possible

## Tips for Success

### Time Management
- **Week 1:** Spend time on design. A good GDD saves hours of coding.
- **Weeks 2–3:** Build fast. Aim for a playable game ASAP, even if it's rough.
- **Weeks 4–5:** Polish and polish more. Playtesting and iteration matter.

### Playtesting
- **Playtest early and often.** Show your game to friends at Day 10, 15, and 20.
- **Listen to feedback.** If people say something is confusing, it probably is.
- **Don't get defensive.** Criticism helps you make a better game.
- **Take notes.** Write down what peers say, then decide what to implement.

### Common Pitfalls
- **Scope creep:** You had 10 ideas, but you only have 25 days. FOCUS. Pick ONE core mechanic and nail it.
- **Ignoring playtesting:** Your game feels fine to you, but a fresh player might find it confusing.
- **Waiting until the last minute to export:** Export your build by Day 23. This gives you 2 days to fix any build issues.
- **Skimping on documentation:** The GDD and code comments are graded. Invest time here.

### Debugging Checklist
- [ ] Is the object active in the Hierarchy?
- [ ] Are all Inspector fields assigned?
- [ ] Does the Console show errors?
- [ ] Did you press Ctrl+S to save?
- [ ] Is the script attached to the right object?
- [ ] Are your scenes in Build Settings?
