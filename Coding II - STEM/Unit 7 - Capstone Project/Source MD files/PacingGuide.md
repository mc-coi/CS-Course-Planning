# Unit 7 - Capstone Project: Pacing & Differentiation Guide

## Unit Overview
- **Total days allocated:** 6 days / 2 weeks
- **Core topics covered:** Project planning (requirements, scope, user stories), system design and architecture, incremental implementation, testing and debugging, code refactoring, documentation (docstrings, comments), and professional presentation
- **Prerequisites students need:** Mastery of all Coding 2 STEM units (1–6); ability to write well-structured code; understanding of data structures, OOP, algorithms, file I/O, and APIs

## Unit Structure Overview

This unit is fundamentally different: it's a capstone project, not a skills unit. The "pacing" is more about project phases than daily lessons. Students work on a self-selected or instructor-assigned project that demonstrates semester-long skills.

### Suggested Timeline
- **Days 1–2:** Planning & design (define requirements, design architecture)
- **Days 3–4:** Implementation (code incrementally, test as you go)
- **Day 5:** Refinement (refactor, document, optimize)
- **Day 6:** Presentation (demo and documentation)

This timing is flexible and depends on project scope.

## Project Selection & Scope

### Instructor-Provided Project Ideas (from Unit Practice)
1. **Task/To-Do Manager** — Store tasks (lists/dicts); save to file; filter/sort
2. **Expense Tracker** — Track spending by category; generate reports; visualize with charts
3. **Grade Calculator** — Calculate GPA; track multiple subjects; generate transcript
4. **Quiz Application** — Create quiz questions; track scores; analyze performance
5. **Weather Dashboard** — Fetch from weather API; display forecasts; visualize data
6. **Contact Manager** — Store contacts with details; search; export/import CSV
7. **Habit Tracker** — Track daily habits; visualize streaks; statistics
8. **Movie Database** — Search movies via API; rate/watchlist; data storage
9. **Personal Finance Tracker** — Budget tracker; savings goals; visualization
10. **Study Aid (Flashcard)** — Create flashcards; spaced repetition algorithm
11. **Workout Planner** — Create routines; track progress; visualize improvements
12. **Meal Planner** — Plan meals; generate shopping lists; nutrition tracking
13. **Reading Tracker** — Track books; rate; create reading lists
14. **Chatbot/Discord Bot** — Automated responses; fun commands
15. **Data Analysis Tool** — Process CSV; calculate stats; visualize results

### Student-Proposed Projects
Students can propose their own projects, subject to:
- Must integrate **at least 3 units** from the curriculum (e.g., OOP + File I/O + visualization)
- Must demonstrate **5+ program requirements** (features that code must do)
- Scope must be completable in 6 days

## Week-by-Week Pacing & Phases

### Days 1–2: Planning & Design (Essential)
**Deliverables:**
- Project scope document (problem statement, must-haves vs. nice-to-haves)
- User stories (3–5 describing user perspectives)
- Data structure design (what classes/dicts/lists will you use?)
- Architecture plan (how components interact)

**Activities:**
- Students define requirements clearly
- Design data structures before coding
- Create TODO list of features to implement
- Identify which units they'll use and how

**If ahead of pace:**
- Add wireframes or flowcharts
- Detailed technical specifications
- Design patterns consideration
- Risk analysis and mitigation

**If behind pace:**
- Provide planning templates with blanks to fill
- Simplify scope: Reduce to 3 must-have features
- Use suggested project ideas; reduce design phase

**Differentiation:**
- **Struggling:** Provide 50% of design document; scaffold heavily
- **On-track:** Provide template; complete independently
- **Advanced:** No template; full freedom with constraints

### Days 3–4: Incremental Implementation (Essential)
**Deliverables:**
- Working code for all planned features
- Functions/classes for each major component
- At least one test/demo of each feature
- Code written for the 50% or more of planned features by Day 3

**Checkpoints:**
- Day 3: Checkpoint—50% of features working; code review
- Day 4: Final implementation; all features attempted

**Activities:**
- Code incrementally; test as you go
- Use version control (git) if possible
- Debug and fix issues immediately
- Refactor for clarity as you write

**If ahead of pace:**
- Add bonus features (advanced sorting, extra API calls, animations)
- Optimize performance
- Add edge case handling

**If behind pace:**
- Reduce scope further; cut lowest-priority features
- Pair program with peer or instructor
- Use provided helper functions for complex logic

**Differentiation:**
- **Struggling:** Pair programming, frequent check-ins, provided helper functions
- **On-track:** Independent coding, daily check-ins, self-debugging
- **Advanced:** Solo coding, complex features, peer mentoring

### Day 5: Refinement & Documentation (Highly Flexible)
**Deliverables:**
- Refactored code (improved clarity and efficiency)
- Complete docstrings and comments
- README.md explaining how to use the program
- Bug fixes from Day 4

**Activities:**
- Refactor for readability
- Add docstrings to all functions/methods
- Add comments for complex logic
- Test edge cases and fix remaining bugs
- Create user guide or README

**If ahead of pace:**
- Optimize algorithms (implement sorting/searching from Unit 5)
- Add logging and error handling
- Create comprehensive testing suite
- Performance profiling

**If behind pace:**
- Focus on code that works over perfect documentation
- Minimal docstrings (at least docstring per function)
- README: Just basic usage instructions
- Skip optimization

**Differentiation:**
- **Struggling:** Provided templates for docstrings; light refactoring
- **On-track:** Standard refactoring and documentation
- **Advanced:** Deep refactoring, professional documentation, code review checklist

### Day 6: Presentation & Submission (Essential)
**Deliverables:**
- Working demonstration (show code and output)
- Presentation (5–10 minutes) or demonstration video
- Project submission with all code and documentation
- Self-evaluation/reflection on learning

**Activities:**
- Prepare demo with example data
- Create presentation slides (optional)
- Practice demo to show off features
- Write reflection on what was learned
- Submit final code and documentation

**Presentation Rubric:**
- Code runs without errors (30%)
- Demonstrates 5+ features/requirements (30%)
- Code is organized and readable (20%)
- Presentation is clear and professional (20%)

## Differentiation Strategies

### For Struggling Students
**Scaffolding tips:**
- Provide detailed planning templates
- Offer choice of instructor-designed projects (not blank canvas)
- Pair with stronger student for implementation phase
- Provide partial code (helper functions, class structures)
- Shorter project scope (3 must-haves instead of 5+)
- More frequent check-ins (every 1–2 days)

**Project scope for struggling students:**
- 1 data structure (list of dicts or simple class)
- 1 file I/O operation (read or write)
- 3 core features (essential functionalities)
- Minimal visualization or external libraries

**Example:** Task Manager with 3 core features:
1. Add task (store in list)
2. View all tasks (print list)
3. Mark complete (edit task status)
- Optional: Save to file (Unit 4)

**Check-in schedule:**
- Day 1: Approve design
- Day 2: Meet to discuss any blockers
- Day 3: Code review at 50%
- Day 4: Final check before submission
- Day 6: Demo and presentation

### For On-Track Students
**Project scope for on-track students:**
- 2–3 data structures or classes
- File I/O (read and write, or CSV)
- 5+ core features
- Error handling for common issues
- Basic documentation

**Example:** Weather Dashboard
1. Fetch data from API (Unit 6)
2. Parse and process data (Unit 4)
3. Display in organized format
4. Save to file for history
5. Create visualization (Unit 6)

**Check-in schedule:**
- Day 1: Quick approval
- Day 3: Code review at 50%
- Day 5: Final refinement check
- Day 6: Demo and presentation

### For Advanced Students
**Project scope for advanced students:**
- 3+ interrelated classes or complex data structures
- Multiple file I/O operations (read/write, CSV, JSON)
- 7+ features with edge cases handled
- API integration or external library use
- Optimization applied (sorting, searching from Unit 5)
- Comprehensive documentation and testing

**Example:** Expense Tracker with Dashboard
1. Add/edit/delete expenses with categories (OOP)
2. Generate reports (Unit 5 sorting)
3. Visualize spending (Unit 6 matplotlib)
4. Calculate statistics and budgets (Unit 3 algorithms)
5. Save/load from CSV and JSON (Unit 4)
6. Search by category, date, amount (Unit 5 searching)
7. Detect anomalies (advanced analysis)

**Enhancement ideas for advanced students:**
- Machine learning (simple predictions)
- Real-time data fetching (Unit 6 APIs)
- Web interface (Flask, Streamlit)
- Database integration (SQLite)
- Automated testing suite

## Suggested Checkpoints & Rubric

### Checkpoint 1: Day 1 Planning
- [ ] Scope document complete (problem, requirements, must-haves)
- [ ] 3–5 user stories written
- [ ] Data structure design planned
- [ ] Feature list with priority
- [ ] Realistic timeline for Days 3–4

### Checkpoint 2: Day 3 Implementation (50%)
- [ ] 50%+ of planned features implemented
- [ ] Code organized into functions/methods
- [ ] At least 1–2 features fully tested
- [ ] No major syntax errors
- [ ] Documented plan for remaining features

### Checkpoint 3: Day 5 Refinement
- [ ] All planned features implemented
- [ ] Docstrings added to all functions
- [ ] Code refactored for clarity
- [ ] README or usage guide exists
- [ ] Bugs from Day 4 fixed

### Final Rubric (Day 6)

| Criteria | Excellent (90%+) | Good (80-89%) | Satisfactory (70-79%) | Needs Work (<70%) |
|----------|---|---|---|---|
| **Functionality** | All 5+ features work perfectly; edge cases handled | 5+ features work; minor bugs | 4 features work; some bugs | <4 features; major bugs |
| **Code Quality** | Well-organized, efficient, readable, DRY principle | Organized, mostly efficient, readable | Some organization, some redundancy | Disorganized, hard to follow |
| **OOP/Design** | Appropriate classes/structures; good design | Classes/structures used well | Basic structures work | Minimal/poor design |
| **Documentation** | Complete docstrings, comments, README | Good docstrings, most comments | Basic docstrings, minimal comments | Little/no documentation |
| **Presentation** | Clear, confident demo; explains design decisions | Clear demo, good explanation | Shows all features, basic explanation | Confusing demo or missing features |

## STEM-Specific Notes

**Important:** Capstone unit is about integration and reflection, not introducing new concepts. Timing must account for:
- Testing and debugging (always takes longer than expected)
- Scope creep (students add features; enforce must-have priorities)
- Varying project times (some finish by Day 5; others need all 6 days)

### Time Budgeting
- **Planning (Days 1–2):** 20% of time
- **Implementation (Days 3–4):** 50% of time
- **Refinement (Day 5):** 20% of time
- **Presentation (Day 6):** 10% of time

### When to intervene
- **Day 3:** If student at <25% of planned features, reduce scope immediately
- **Day 4:** If major bugs, provide help or extend presentation to Day 7
- **Day 5:** If not ready, allow incomplete submission with documentation of what was learned

### Handling diverse finishing times
- **Early finishers (Day 4):** Add bonus features, optimize code, peer mentor
- **On-time (Day 5):** Standard timeline
- **Late finishers (after Day 6):** Partial credit; document what was learned even if incomplete

## Project Outcomes & Learning Goals

Students should demonstrate:
1. **Planning ability:** Can translate ideas into requirements and design
2. **Implementation:** Can write code that meets requirements
3. **Debugging:** Can identify and fix problems
4. **Communication:** Can explain design and present code
5. **Integration:** Can use multiple units together (OOP + File I/O, APIs + Visualization, etc.)
6. **Reflection:** Can identify what they learned and what was hard

## Connection to Coding 2 Curriculum

Capstone should integrate concepts from multiple units:

| Unit | Integration Example |
|------|---|
| **Unit 1: Data Structures** | Use lists, dicts to organize project data |
| **Unit 2: OOP** | Create classes for project entities (Task, Expense, Contact, etc.) |
| **Unit 3: Advanced Functions** | Use comprehensions, lambda, recursion where appropriate |
| **Unit 4: File I/O** | Save/load project data from CSV or JSON |
| **Unit 5: Sorting/Searching** | Sort results, search by criteria |
| **Unit 6: APIs/Libraries** | Fetch real data, visualize results |

A strong capstone project touches 4–6 of these units.

## Grading Suggestions

- **Planning (10%):** Clear requirements, realistic design
- **Implementation (40%):** Working features, code quality, error handling
- **Documentation (20%):** Docstrings, comments, README
- **Presentation (20%):** Clear demo, professional presentation
- **Reflection (10%):** Learning goals met, honest assessment of work

**Total possible: 100%**
